# ChatGPTレビュー依頼: 内容選択と使用素材確認の再設計

作成日: 2026-06-27

## レビューしてほしいこと

zev2 の切り抜き候補決定プロセスを、旧「AIがテーマ候補を出して人間がテーマを選ぶ」から、新しい「AIが何があるかを説明し、人間が面白そうな内容を選び、AIが具体素材を探し、人間が素材で進めてよいか確認する」形へ変更した。

この設計が、人間のコントロールを失わず、AIに無理な面白さ判定をさせず、Gemini APIを動画読解に集中させる流れになっているか確認してほしい。

## 新しい工程理解

AI工程は7つのまま。
人間の判断は、AI工程とは別の人間確認工程として扱う。

1. 動画取り込み
2. STT
3. 内容候補整理
4. 使用素材構成
5. 演出作成
6. 微調整
7. 動画生成

追加または再定義した人間確認工程:

- 内容選択: 内容候補整理の後。人間が面白そうな出来事、話題、反応を選ぶ。
- 使用素材確認: 使用素材構成の後。AIが見つけた素材で演出作成へ進んでよいか判断する。
- 動画生成前確認: 微調整の後。確認用動画を作ってよいか判断する。
- 生成後レビュー: 完成動画を見て、採用するか、どこから作り直すか判断する。

## 役割分担

- AIは、文字起こし内に何があるかを説明する。
- 人間は、どの内容が面白そうかを判断する。
- AIは、選ばれた内容に関係する具体的な発話箇所を探し、使用素材構成案にする。
- 人間は、見つかった素材で動画にできそうか確認する。
- Gemini APIは、確認済みの動画断片を読む演出作成で使う。

人間に発話ID、秒数、具体素材の細かい採否は選ばせない。
Gemini APIに、文字起こしだけを読ませて面白いテーマを選ばせる流れにはしない。

## 今回ローカルで変更したこと

- `docs/zev2-flow-contract.md` を新方針の正本として更新した。
- `docs/artifacts/README.md`、`docs/review-brief.md`、`docs/current-implementation.md`、`docs/order.md`、`docs/runner-dry-run.md` などを新方針へ更新した。
- 設定名を `themeExploration` から `contentDiscovery` に変更した。
- `contentDiscovery.mode` は `fixed` または `transcript` にした。
- 内容候補整理ではGemini APIを呼ばず、固定データまたは文字起こし整理で内容候補を作るようにした。
- 画面上の実行設定表示を「テーマ探索」から「内容候補整理」へ変更した。
- runnerの工程表示名を「テーマ候補作成」から「内容候補整理」、「複数箇所構成」から「使用素材構成」へ変更した。
- 内容候補整理、使用素材構成、微調整の完了時に人間確認を作り、承認されるまで次工程へ進まないようにした。
- UIに内容選択、使用素材確認、動画生成前確認の確認カードを追加した。

## まだ未実装の重要点

- 内容選択と使用素材確認は最小UIで操作できるが、表示情報の読みやすさはまだ改善余地がある。
- 内部型名やファイル名には `theme_json`、`selectedThemeId`、`propose_clip_themes` など旧名が残っている。
- 現時点では、後方互換分岐を増やさず、表示と設定と処理の意味を先に寄せた状態。

## 確認済み

- `pnpm run type-check`
- `pnpm run build:client`
- `pnpm run scenario:agent`
- `contentDiscovery.mode = transcript` の単体確認

`pnpm run scenario:agent` では、内容選択、使用素材確認、動画生成前確認でAI工程が止まり、テスト側で承認すると後続工程へ進み、音声付きMP4生成まで到達した。

## レビュー観点

1. 新しい内容選択と使用素材確認の2段階は、人間の判断負担とAIの役割分担として妥当か。
2. 人間に具体素材を選ばせない方針は、UXとして正しいか。
3. Gemini APIを演出作成の動画読解へ寄せ、内容候補整理では使わない方針は妥当か。
4. AI工程を増やさず、人間確認工程で止める整理に問題はないか。
5. 次に実装するなら、確認カードの読みやすさ、作り直し範囲、内部型名の整理のどれを優先すべきか。
