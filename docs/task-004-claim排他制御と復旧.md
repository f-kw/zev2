# task-004 claim排他制御と復旧

作成日: 2026-05-31

## 目的

- 複数AIエージェントが同じ作業を同時に処理しないようにする。
- AIエージェント停止や通信失敗で `running` のまま残った作業を復旧できるようにする。
- 作業取得の所有者、開始時刻、期限切れ条件をAPI上で扱えるようにする。

## 作業スコープ

今回やること:

- `claim` 時にAIエージェント識別子を受け取れるようにする。
- 作業状態へ取得者、取得時刻、最終更新時刻を保存する。
- 取得済み作業を別AIエージェントが取得できないことをテストする。
- 期限切れした取得済み作業を再取得可能にするAPIまたは判定を追加する。
- `complete` と `fail` は取得者が一致する場合だけ成功するようにする。
- 期限切れ、所有者不一致、前工程未完了のエラーを意味で返す。

今回やらないこと:

- 認証を実装しない。
- 分散ロック基盤を導入しない。
- 実STT、実LLM、動画生成を実行しない。
- UIから強制復旧する画面はこのタスクでは作らない。

## 完了条件

- 同じ作業に対する二重claimが防止される。
- 所有者が違う `complete` と `fail` が拒否される。
- 期限切れした作業を復旧対象として扱える。
- 復旧対象の作業が次回取得可能になる。
- 状態確認APIで、取得者と復旧が必要な状態を説明できる。
- `pnpm run type-check` と対象テストが成功する。

## 実装済み確認

- `claim` は取得者IDを必須にし、取得時刻、最終更新時刻、任意の期限時刻を作業状態へ保存する。
- `complete` と `fail` は、`claim` した取得者IDと一致しない場合に拒否する。
- 期限切れした `running` 作業は、状態確認または次作業取得時に `queued` または `waiting` へ戻し、復旧理由と前回取得者を状態へ残す。
- 期限切れ時間は `config/runtime.jsonc` の `agentClaim.ttlMilliseconds` またはAPIの `expiresAt` で明示する。コード内に固定の期限時間は置かない。
