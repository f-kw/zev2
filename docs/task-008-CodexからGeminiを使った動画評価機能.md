# task-008 CodexからGeminiを使った動画評価機能

作成日: 2026-05-31

## 目的

- CodexがGeminiを使って動画を評価し、評価結果をzev2のAIエージェントAPIへ返せるようにする。
- Gemini評価を、STT、LLM候補生成、動画生成とは別の「評価工程」として扱う。
- 評価結果を人間が確認でき、後続工程が参照できる成果物として残す。

## 位置付け

このタスクは実処理接続に近いため、次のタスクが先に完了していることを前提にする。

- task-001 最小E2E dry-run実行
- task-002 AIエージェントAPI契約テストと仕様固定
- task-004 claim排他制御と復旧
- task-005 成果物保存とFileRef実体管理
- task-006 AI操作ログと監査履歴
- task-009 セキュリティ境界と実処理接続方針

## 作業スコープ

今回やること:

- 既に作業種別として用意した `gemini_video_review` の処理境界を具体化する。
- CodexがGeminiへ渡す動画入力の形式を決める。
- Geminiへ送る評価依頼文のテンプレートを作る。
- 評価結果のJSON形式を決める。
- 評価結果を `FileRef` として保存し、後工程から参照できるようにする。
- 評価結果をUIで短く確認できるようにする。
- Gemini操作はCodexから行い、backend内ではGeminiを直接実行しない境界を固定する。
- 評価の観点は明示された項目だけにし、独自の係数や重み付けを使わない。

今回やらないこと:

- backendにGemini APIキーを保存しない。
- backendからGeminiを直接呼ばない。
- STT、LLM候補生成、動画生成をこのタスクで実装しない。
- Gemini評価結果だけで自動採用や自動投稿をしない。
- 評価基準に勝手な係数、点数重み、閾値を追加しない。
- 秘密情報、認証情報、個人情報をGeminiへ送らない。

## 評価観点案

初期案では、次のような人間が読める観点だけを扱う。

- 内容がショート動画として成立しているか。
- 冒頭に見る理由があるか。
- 切り抜き範囲が自然か。
- テロップや説明が必要な箇所はどこか。
- 誤解、危険表現、文脈不足がないか。
- 人間確認が必要な懸念点は何か。

点数化する場合も、ユーザーが明示した評価段階だけを使う。
CodexやGeminiが独自に係数や重みを作って総合点を計算しない。

## 成果物形式案

Gemini評価結果は、状態ファイルへ全文を埋め込まず、成果物として保存する。

```json
{
  "summary": "動画評価の短い要約",
  "usable": true,
  "reviewRequired": true,
  "findings": [
    {
      "kind": "context_missing",
      "meaning": "前提説明が不足している",
      "timeRange": "00:12-00:18"
    }
  ],
  "nextAction": "人間が該当箇所を確認する"
}
```

これは形式案であり、実装時にAPI契約テストと合わせて固定する。
