# zev2 コントロール維持ログ

このログは、AIコーディングで開発物が人間のコントロールを離れないようにするための作業記録です。
コード差分ではなく、処理の意味、判断理由、確認結果を短く残します。

## 2026-06-24 成果物仕様と検証を先に固定する

目的:

- 工程ごとの入力、処理、成果物を人間が追えるようにする。
- AIや固定データが何を作ったのかを成果物の形で確認できるようにする。
- コード分割を急ぐ前に、工程間で渡るデータの正しさを固定する。

外部レビューの要点:

- 直近の疎結合化は方向性として妥当。
- ただし、まだ実処理本体が巨大なrunner内に残っている。
- 次に急ぐべきは全面分割ではなく、成果物仕様、サンプル、実行時検証。
- 動画生成や微調整の分割は外部処理やUIゲートが絡むため後回し。

今回の作業方針:

- 成果物ごとの人間向け仕様を追加する。
- 成果物サンプルを追加する。
- runnerで成果物を読んだ時、書く前に、最低限の構造を検証する。
- 固定確認と通常実行が混ざって見えないようにする。

実施結果:

- `docs/artifacts/README.md` に成果物ごとの入力、作る工程、次に読む工程、最低条件を追加した。
- `docs/artifacts/samples/` に、文字起こし、テーマ候補、複数箇所構成案、編集案、調整結果の短いサンプルを追加した。
- `runner/src/workflow-artifact-validation.ts` に、成果物の最低条件を確認する処理を追加した。
- `runner/src/workflow-step-builders.ts` で、前工程成果物を読んだ時と成果物を書く前に検証するようにした。

確認結果:

- 型検査は成功した。
- 7工程、固定確認、成果物連鎖、固定確認なしのSTT失敗ケースを含むシナリオテストは成功した。

この区切りでできるようになったこと:

- 壊れた成果物を後続工程が読んで曖昧に失敗する前に、成果物を作った工程または読んだ工程で止められる。
- 人間が成果物の意味と最低条件を、コードではなく文書とサンプルで確認できる。
