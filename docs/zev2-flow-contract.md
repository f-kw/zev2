# zev2 7工程フロー契約

作成日: 2026-06-24

この文書は、zev2 の実装で迷わないための固定仕様です。
既存文書や実装コメントと違う場合は、この文書を優先して整理します。

成果物ごとの詳しい人間向け仕様とサンプルは [成果物仕様](artifacts/README.md) に置きます。

## 目的

最初に固定する目的は、AI品質や動画品質ではなく、7工程キュー、作り直し、UI表示、runner実行、テストを同じ前提にそろえることです。

この契約で守ること:

- AI工程は7工程に固定する。
- 人間の選択や承認は、AI工程を増やさず、人間確認工程として扱う。
- 作り直しは元編集を壊さず、新しい編集コピーで再開する。
- backend は外部処理を実行せず、依頼、工程、状態、成果物参照だけを管理する。
- 成果物本体、STT全文、LLM全文、動画本体を状態ファイルへ埋め込まない。
- 外部接続やAI出力が使えない場合、成功したふりをしない。

## 用語

| 用語 | 意味 |
| --- | --- |
| 依頼 | ユーザーが作りたい動画の目的、入力動画、設定をまとめたもの |
| 編集コピー | 1つの依頼から作られる実行単位。作り直すたびに新しい編集コピーを作る |
| 現在の編集コピー | UIが進捗と成果物を見る対象。runnerもこの編集コピーだけを実行対象にする |
| 工程 | runner が実行する7種類の作業 |
| 人間確認工程 | ユーザーの内容選択、使用素材確認、動画生成前確認。runnerが実行するAI工程ではないが、全体フローでは必須の工程 |
| 成果物参照 | JSONやMP4本体ではなく、保存先、種別、保存ファイル名、バイト数、SHA-256を指す参照 |
| 固定データ確認 | STTやGemini APIを使わず、固定成果物で状態遷移を確認する開発用モード |

UIで使う言葉は次に絞る。

| UI表示 | 意味 |
| --- | --- |
| 内容候補 | 文字起こし内にある出来事、話題、反応の説明 |
| 使う場面 | AIが見つけた、動画に入れられそうな発話や動画断片 |
| 流れ | 使う場面をどう並べると動画として成立しそうか |
| 見せ方 | 画面枠、テロップ、強調などの演出 |

「動画に入る話」「AIの判断」「つなぎ方」は、ユーザー向けの主表示では使わない。
「素材を選ぶ」という表現も、ユーザー向けの主導線では使わない。人間が選ぶのは具体的な発話IDや時間範囲ではなく、面白そうな内容と、見つかった素材で進めてよいかの判断です。

## 7工程

AI工程は次の7つだけです。
内容選択、使用素材確認、動画生成前確認は、runnerが実行する工程ではなく、人間確認工程です。

| 順番 | 工程 | 入力 | 処理 | 成果物 | 次へ進む条件 |
| ---: | --- | --- | --- | --- | --- |
| 1 | 動画取り込み | 依頼に入っている動画の場所 | 対象動画を取得または参照し、後続工程が読める状態にする | 入力動画の参照 | 入力動画の参照を保存できた |
| 2 | STT | 入力動画の参照 | 音声を文字起こしし、発話ごとのID、開始、終了、話者、本文を作る | 文字起こし | 文字起こしを保存できた |
| 3 | 内容候補整理 | 文字起こし | 文字起こし内に何があるかを、ユーザーが面白そうか判断できる候補として説明する | 内容候補 | 内容候補を保存できた |
| 4 | 使用素材構成 | ユーザーが選んだ内容候補と文字起こし | 選ばれた内容に関係する複数の発話箇所を探し、動画として成立しそうな流れを作る | 使用素材の構成案 | 構成案を保存し、使用素材確認へ進められる |
| 5 | 演出作成 | 確認済みの使用素材構成案と動画断片 | Gemini APIで動画断片を読み、各断片の画面枠、テロップ位置、見せ方を決める | 編集案 | 編集案を保存できた |
| 6 | 微調整 | 編集案 | 動画生成で使える形へ整え、必要な修正を反映する | 生成用の調整結果 | 調整結果を保存できた |
| 7 | 動画生成 | 編集案と調整結果 | 複数箇所を切り出して連結し、テロップを重ね、音声付きMP4を作る | 完成動画 | MP4を保存できた |

### 工程ごとの禁止事項

- 内容候補整理で、AIが面白さや採用可否を決めない。
- 内容候補整理で、具体的な使用素材、構成案、テロップ案を混ぜない。
- 使用素材構成で、画面枠やテロップ装飾を決めない。
- 使用素材構成で、人間に発話IDや秒数を選ばせない。
- 演出作成で、使用素材確認済みの場面を勝手に増減しない。
- 微調整で、内容候補や使う場面を再選定しない。
- 動画生成で、意味判断やAI判断をしない。与えられた調整結果を実行する。

## 内容候補と使用素材確認

テーマ選択は、旧設計では1段階だった。
新設計では、人間が面白さを判断し、AIが具体素材を探す仕事を分ける。

| 段階 | 担当 | 処理の意味 | やらないこと |
| --- | --- | --- | --- |
| 内容候補整理 | runner | 文字起こしを読み、出来事、話題、反応として何があるかを説明する | 面白さの採点、採用判断、動画断片の確定 |
| 内容選択 | 人間 | 面白そうか、この方向で作りたいかを選ぶ | 発話ID、秒数、素材の細かい採否を選ぶ |
| 使用素材構成 | runner | 選ばれた内容に関係する発話箇所を複数探し、動画としての流れを作る | 画面枠、テロップ位置、映像演出の判断 |
| 使用素材確認 | 人間 | AIが見つけた素材で動画にできそうかを判断する | 細かい秒数編集、画面レイアウト指定 |
| 演出作成 | runner + Gemini API | 確認済みの動画断片をGemini APIへ渡し、映像を読んで見せ方を作る | 内容候補の再選定 |

Gemini APIを使う主な理由は、動画断片を読ませるためです。
文字起こしだけで行う内容候補整理や使用素材構成では、Gemini APIである必然性は低い。
そのため、Gemini APIを使う場所は演出作成に寄せる。

## 人間確認工程

人間確認工程はAI作業キューに増やさない。
ただし、後続工程が進む条件として必ず扱う。
工程4の後には必ず使用素材確認を置き、AIが見つけた素材を説明してから、人間がその素材で進められるか判断する。

| 人間確認工程 | 位置 | ユーザーが決めること | 後続工程 |
| --- | --- | --- | --- |
| 内容選択 | 工程3の後 | どの出来事、話題、反応が面白そうか | 工程4以降 |
| 使用素材確認 | 工程4の後 | AIが見つけた素材で動画にできそうか | 工程5以降 |
| 動画生成前確認 | 工程6の後 | 調整結果で確認用動画を作ってよいか | 工程7 |
| 生成後レビュー | 工程7の後 | 採用するか、どこから作り直すか | 新しい編集コピー |

固定データ確認では、人間確認工程の入力をテストデータとして与えてよい。
ただし、UIやログでは「自動で与えたテスト用判断」と分かるようにし、APIキー不足や外部接続失敗を成功のように見せてはいけない。
runnerで固定データ確認を使う場合は、`config/runtime.jsonc` またはrunner起動時の環境変数で、対象工程の実行方法を `fixed` にする。
この指定がない通常実行では、STT接続やGemini API接続がなければ該当工程を失敗にする。
固定データ確認で作る文字起こし、内容候補、編集案は、通常のSTT結果やGemini API結果とは別のサンプル成果物として保存する。

## 作り直し

作り直しは、既存の編集コピーを書き換えない。
必ず新しい編集コピーを作り、現在の編集コピーを新しいものへ切り替える。

| UI表示 | 意味 | 引き継ぐ成果物 | 作り直す工程 |
| --- | --- | --- | --- |
| 内容候補から作り直す | 文字起こしは使い、内容候補から作り直す | 工程1、工程2 | 工程3から工程7 |
| 内容を選び直す | 既存の内容候補から別の内容を選ぶ | 工程1、工程2、工程3 | 内容選択、工程4から工程7 |
| 同じ内容で素材を探し直す | 内容選択は使い、使用素材構成から作り直す | 工程1から工程3、内容選択 | 工程4から工程7 |
| 演出作成前から作り直す | 構成案は使い、見せ方から作り直す | 工程1から工程4 | 工程5から工程7 |
| 微調整前から作り直す | 編集案は使い、生成用の調整から作り直す | 工程1から工程5 | 工程6から工程7 |

「テーマ決定前から作り直す」という表示は曖昧なので使わない。
内容候補そのものを作り直すのか、既存内容候補から選び直すのか、同じ内容で素材だけ探し直すのかを分ける。

### 作り直しの不変条件

- 元の編集コピーの工程状態、成果物参照、完成動画は変更しない。
- 新しい編集コピーには、再開地点より前の成果物参照だけをコピーする。
- 再開地点以降の工程は、新しい編集コピー側で未実行にする。
- UIは新しい編集コピーを現在の進捗対象にする。
- runnerは現在の編集コピー以外の未実行工程を拾わない。
- 古い編集コピーは閲覧専用として扱う。

## 状態

工程状態は次の意味で使う。

| 状態 | 意味 |
| --- | --- |
| 作業待ち | 前提条件がそろえばrunnerが取得できる |
| 前工程待ち | 前工程または人間確認工程の完了待ち |
| 実行中 | runnerが担当中 |
| 完了 | 成果物参照または完了意味を保存済み |
| 失敗 | 失敗理由を保存済み。後続工程は進まない |
| 対象外 | 作り直しによって古い編集コピー側の実行対象から外れた |

失敗を成功扱いにしない。
Gemini APIキー、STT接続、動画取得、動画生成に必要な情報がない場合は、その工程を失敗にし、ユーザーが何を直せばよいか分かる文で表示する。

## 責務境界

| 部品 | 担当 | 担当しないこと |
| --- | --- | --- |
| UI | ユーザー入力、内容選択、使用素材確認、作り直し指示、進捗表示 | STT、Gemini API、動画生成、状態の推測補完 |
| backend | 依頼、編集コピー、7工程キュー、状態、成果物参照の管理 | STT、LLM、Gemini API、ffmpeg、Remotionの実行 |
| runner | 次工程の取得、担当中化、実処理、完了または失敗報告 | ユーザー選択の代行、UI判断、作り直し判断 |
| AIエージェント | 内容候補整理、使用素材構成、Gemini APIへ渡す入力作成、応答検証、入出力ログ保存 | backend状態の直接書き換え、動画生成の実行 |
| STT | 発話ID、開始、終了、話者、本文の生成 | 内容候補、使用素材、演出の判断 |
| Gemini API | 確認済み動画断片を読んだ演出案、表示候補などの構造化出力 | 内容候補整理、ファイル保存、状態管理 |
| Remotion | テロップとレイアウトの合成 | どの発話を使うかの判断、どの内容が面白いかの判断 |
| ffmpeg | 切り出し、連結、音声付きMP4生成、検証 | テロップ設計、演出判断 |

## コード共通化ルール

第三者向けコメント:

このルールは、単にきれいなコードを書くためのものではありません。AIコーディングで開発速度が上がるほど、同じ意味の処理が少しずつ違う形で増え、人間が全体を追えなくなる危険があります。だから、ID検索、状態判定、成果物参照の読み書きのような一般的な繰り返し処理は共通化します。一方で、どのテーマを選ぶか、どの発話を使うか、どこから作り直すかのような zev2 固有の判断は、共通関数に隠さず、工程と成果物として読める形に残します。

AIコーディングで開発物が人間のコントロールを離れないように、既に実装されている繰り返し処理は共通化する。
目的は、コード量を減らし、読む場所を減らし、同じ意味の処理が複数箇所で少しずつ違う動きをする状態を避けることです。

基本方針:

- コード量は少なくする。
- 工程ごとの実装は独立させる。
- データ構造は人間が完全に理解できる形にする。
- コードで隠してよいのは、どのアプリにもある機械的な処理だけにする。
- zev2固有の判断は、共通化で隠さず、工程と成果物の形で見えるようにする。

共通化してよいもの:

- IDで対象を探す処理。
- 状態が指定した意味に含まれるか判定する処理。
- 工程一覧から対象工程を探す処理。
- 前工程の完了確認。
- 成果物参照を作る処理。
- 成果物参照から保存先を読む処理。
- 日時更新。
- API応答とエラー表示の整形。
- 配列の重複除去、並び替え、絞り込み。
- 入力値の空チェック、必須項目チェック。

共通化してはいけないもの:

- どの内容候補を出すか。
- どの発話を使うか。
- どの流れで見せるか。
- どの見せ方を選ぶか。
- どの工程から作り直すか。
- 何を主表示にするか。
- Gemini APIへ何を判断させるか。
- テロップをどこで切るか。

判断基準:

- どのアプリにもある機械的な処理なら、実装時に自動で共通化してよい。
- zev2固有の業務判断を含む処理は、共通関数に隠さない。
- 共通化で処理の意味が読みにくくなる場合は、共通化しない。
- 共通化する場合は、関数名ではなく処理の意味で説明できる単位にする。
- 1つの共通関数が複数工程の業務判断を持ち始めたら、分割する。

## 工程ごとの疎結合ルール

第三者向けコメント:

このルールは、工程を細かく分けたいから置いているのではありません。人間が「この工程は何を読み、何を作り、次へ何を渡すのか」をコード全体を読まずに確認できるようにするためです。工程同士が密接に絡むと、1つの不具合や仕様変更を理解するためにUI、backend、runner、Gemini応答、動画生成まで追う必要が出ます。その状態では、AIが実装を先へ進めるほど人間の制御が弱くなります。各工程は決められた成果物を受け取り、自分の成果物だけを作り、工程をまたぐ判断は成果物として保存してから渡します。

工程は疎結合にする。
これは、工程を細かく分けるためではなく、コードを完璧に読まなくても人間が工程単位で処理を理解できるようにするためです。

密結合になると、ある工程の意味を理解するために、別工程、UI、runner、AI出力、動画生成までコードを追う必要が出る。
その状態ではAIコーディングが進むほど人間が取り残される。

疎結合の目的:

- 人間が工程単位で理解できる。
- 人間が工程単位で止められる。
- 人間が工程単位で作り直せる。
- 壊れた時に、どの工程の入力、処理、成果物が悪いかを切り分けられる。
- AIが工程をまたいで勝手に判断を混ぜることを防ぐ。

疎結合の成立条件:

- 各工程は、決められた入力成果物だけを読む。
- 各工程は、自分の成果物だけを作る。
- 後続工程は、前工程の内部処理ではなく成果物参照を読む。
- ある工程が、別工程の未公開の内部データや一時状態を直接読まない。
- ある工程が、別工程の成果物を直接書き換えない。
- 工程をまたぐ判断は、成果物として保存してから渡す。
- 失敗時は、その工程で止める。後続工程で推測補完しない。

工程間で渡してよいもの:

- 入力動画の参照。
- 文字起こしの参照。
- 内容候補の参照。
- ユーザーが選んだ内容候補。
- 使用素材の構成案。
- 編集案。
- 生成用の調整結果。
- 完成動画の参照。
- AI入出力ログの参照。

工程間で渡してはいけないもの:

- 工程内だけで使う一時ファイルのパス。
- 画面表示の都合だけで作ったUI状態。
- Gemini APIの未検証応答。
- STTやLLMの巨大な生データ本文。
- 失敗した工程を成功に見せる代替データ。
- 別工程の内部事情に依存したフラグ。

実装判断:

- 1工程だけ差し替えても、前後工程の契約が変わらない形を目指す。
- 固定データ確認と実処理確認は、同じ成果物形式で後続工程へ渡す。
- 工程をまたぐ便利関数を作る場合は、業務判断ではなく成果物の読み書きに限定する。
- UIは現在の編集コピーと成果物参照を見る。工程内部の一時状態を直接見ない。

## データ構造理解ルール

コードは少なくし、工程実装は独立させる。
一方で、データ構造は人間が完全に理解している必要がある。

理由:

- AIが何を入力として受け取ったかを人間が追えるようにするため。
- AIが何を出力したかを人間が確認できるようにするため。
- 作り直し時に、何を引き継ぎ、何を捨てるか判断できるようにするため。
- 失敗時に、コード全体ではなく成果物の形から原因を切り分けられるようにするため。
- データ構造が曖昧だと、AI出力の検証もUI表示も作り直しも不安定になるため。

固定すること:

- 7工程それぞれの入力と成果物の構造を、人間が説明できる形にする。
- 成果物は、工程、意味、入力参照、出力参照、検証結果を持つ。
- AI出力は、必ず固定された成果物構造へ落としてから次工程へ渡す。
- 固定データ確認と実処理確認は、同じ成果物構造を使う。
- UIは、成果物構造から人間向け要約を表示する。
- 成果物構造が人間に説明できない場合、その実装は進めない。

禁止すること:

- データ構造を「とりあえずJSON」として増やす。
- 1つの成果物に内容候補、使用素材構成案、演出案、生成用データを混ぜる。
- 後続工程が、前工程の内部実装を知っている前提でデータを読む。
- AI応答を検証せずにそのまま次工程へ渡す。
- UI都合の表示データを工程間の正本にする。
- 人間が説明できない巨大な成果物構造を正本にする。

人間が握る正本:

- 工程。
- 成果物。
- データ構造。
- 状態遷移。
- 作り直し時の引き継ぎ範囲。

## AI入出力ログ

AIの判断をブラックボックスにしない。
工程3、工程4、工程5、表示候補選択などでAIを使う場合は、次を成果物参照として残す。

- どの工程で使ったか。
- どのモデルを使ったか。
- 入力に使った成果物参照。
- 出力されたJSONまたは判断結果の参照。
- 検証に通ったか。
- 検証に失敗した場合の理由。

秘密情報、APIキー、巨大な動画本体はログへ残さない。

## テロップと発話ID

テロップは時間だけで管理しない。
発話IDに紐付ける。

固定すること:

- STTの各発話にIDを持たせる。
- テロップは1つ以上の発話IDを参照する。
- テロップは参照する発話の開始に合わせて出す。
- テロップは参照する発話の終了とともに消す。
- 文の途中で不自然に切る判断は、固定ルールではなくLLM出力で扱う。
- 動画生成前に、参照先の発話IDが存在することを検証する。

テロップの見た目:

- Remotionで作る。
- ImageMagickはテロップ品質の本線にしない。
- ショート動画なので文字を小さくしない。
- 顔には重ねない。
- 座布団あり、座布団なしのどちらも設定データで切り替えられる形にする。
- 美化は後続実装。先にID紐付けと消えるタイミングを固定する。

## 表示枠

最初に扱う表示枠は次の3種類に限定する。

| 表示枠 | 画面 |
| --- | --- |
| 話者のみ | 縦長1枠。話者の顔を必ず入れる |
| 画面と話者 | 上下2枠の横長表示 |
| 話者2人 | 上下2枠の横長表示 |

表示枠の選択理由と入力は保存する。
顔を必ず入れる高度な調整は、状態遷移とテストが固まってから強化する。

## モデル設定

依頼ごとにGeminiモデルを選べるようにする。

| 用途 | モデル |
| --- | --- |
| 品質確認 | Gemini 3.5 Flash |
| 軽い確認 | Gemini 3 Flash Preview |
| 疎通確認 | Gemini 2.5 Flash |

固定すること:

- UIに選択中モデルを表示する。
- 実行成果物に使用モデルを残す。
- 疎通確認やJSON構造確認だけのテストでは、高品質モデルを使わなくてよい。
- モデル指定がない場合は、成功したふりをせず標準モデルを明示して記録する。

## UI主導線

UIはユーザーを疲れさせない。
動画の情報とUIの使い方説明を同列に並べない。

生成後レビューの主導線:

```text
動画を見る
直したい点を書く
作り直し範囲を選ぶ
作り直す
```

工程表示はタブではなく、進捗表示として扱う。
詳細情報、ログ、JSON、長い説明は初期表示で主導線に置かない。

## 最低限のテスト

仕様固定後、次のテストを実装対象にする。

### 状態遷移

- 新規依頼で7工程が作られる。
- 前工程が完了するまで後続工程は実行されない。
- 実行中の工程だけ完了できる。
- 失敗した工程がある場合、後続工程は進まない。
- 失敗した工程を再実行できる。

### 作り直し

- 内容候補から作り直すと、工程1と工程2だけを引き継ぐ。
- 内容を選び直すと、工程1から工程3だけを引き継ぐ。
- 同じ内容で素材を探し直すと、工程1から工程3と内容選択だけを引き継ぐ。
- 演出作成前から作り直すと、工程1から工程4だけを引き継ぐ。
- 微調整前から作り直すと、工程1から工程5だけを引き継ぐ。
- 作り直し後、UIとrunnerが同じ現在の編集コピーを見る。
- 古い編集コピーはrunner対象にならない。

### UI

- 生成後レビューで、動画、修正入力、作り直し範囲、実行ボタンが同じ視界に入る。
- 詳細情報は初期状態で閉じている。
- 作り直し後、新しい編集コピーの進捗へ切り替わる。
- 古い完成動画は消えない。

### AI接続

- Gemini APIの正常応答が構造検証を通る。
- 必須項目がないAI応答は失敗になる。
- 存在しない発話IDを参照するAI応答は失敗になる。
- AI入力、AI出力、使用モデルが成果物参照として残る。

### 動画生成

- 完成MP4に映像がある。
- 完成MP4に音声がある。
- 長さが0ではない。
- 複数箇所がつなぎ順どおりに並ぶ。
- テロップが発話IDに対応して切り替わる。

## 実装順序

この順で進める。

1. この契約を固定する。
2. 状態遷移、7工程キュー、作り直しコピー、現在の編集コピーをテストで固定する。
3. 生成後レビューUIを、動画、修正入力、作り直し範囲中心に整理する。
4. UIの主要シナリオを自動テストにする。
5. STT、Gemini API、Remotion、ffmpegの接続境界をadapterとして整理する。
6. 内容候補整理、内容選択、使用素材構成、使用素材確認を強化する。
7. 演出作成、微調整、動画生成品質を強化する。
8. 表示枠、テロップ美化、モデル設定UIを追加する。

先にやらないこと:

- 実STTの品質改善。
- 内容候補整理でGemini APIへ依存すること。
- Gemini APIによる高度な動画理解。
- 顔を必ず入れる高度な自動調整。
- 美しいテロップの大量バリエーション。
- 複雑なモデル設定UI。
- 詳細ログビューア。

## 危険信号

次が起きたら実装を止めて、仕様またはテストへ戻る。

- 「内容候補」と「使用素材構成案」と「テロップ案」が同じ成果物に混ざる。
- 作り直し後、UIとrunnerが違う編集コピーを見る。
- 失敗した外部接続を固定データで成功したように見せる。
- backendがSTT、Gemini API、ffmpeg、Remotionを直接呼ぶ。
- 1つの関数が複数工程の業務判断をまとめて持つ。
- 後方互換分岐で古い形式を推測補完し始める。
- ユーザーの手動確認がないと主要シナリオの成否が分からない。
