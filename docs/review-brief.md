# zev2 レビュー用 要点

作成日: 2026-05-31
更新日: 2026-06-21

レビュー時はこの文書だけ見る。
詳細仕様やタスク全文は、判断後に必要な部分だけ見る。

## いま作っているもの

人間が依頼を入力し、AIエージェントがAPIで作業を進める基盤。

backendはSTT、LLM、Gemini API、動画生成、完成品レビューを直接実行しない。
実処理はAIエージェントrunnerやCodex側の処理が行い、backendには結果と成果物参照だけを返す。

## 現時点の形

- UI操作は「承認してAIに渡す」の1操作。
- 内部状態では、依頼作成と承認済み作業を分ける。
- AIエージェントは承認済み作業だけをAPIで取得する。
- UIは依頼、テーマ選択、構成と演出確認、生成後レビューに絞る。
- 候補決定は文字起こしからテーマ候補を出し、人間が切り抜きたいテーマを選ぶ流れにする。
- 選ばれたテーマに関係する複数箇所を集め、つなげて構成案を作る。
- runnerで、動画参照、ZEVローカルSTT、Gemini APIによるテーマ候補、Gemini APIによる演出案、MP4生成まで進められる。
- 演出案では、Gemini APIが動画断片ごとに画面へ出す対象を検出し、runnerが「話者のみ」「画面と話者」「話者2人」の画面枠へ合成してから複数断片を連結する。話者のみは縦長1枠、2枠表示は上下の横長枠にする。
- 「承認してAIに渡す」後、backendがdry-run runnerをバックグラウンド起動する。
- 完成品レビュー用のWeb版Gemini接続はまだ動かさない。

## 依頼と承認を内部で分けるメリット

- AIエージェントが未確定の入力を取得しない。
- 人間が実行対象を確定した履歴を残せる。
- 外部API費用や副作用がある処理を、承認済み作業に限定できる。
- 失敗、再実行、差し戻し、監査ログを後から整理しやすい。

UI上は1ボタンでよい。
内部状態として分けることに意味がある。

## UIで確認できるべきもの

- 依頼内容。
- テーマ候補と代表発話。
- 人間が選んだテーマ。
- 選ばれたテーマに関係する複数箇所とつなぎ方。
- 演出案と動画生成前の確認。
- 最新の成果物参照と失敗理由。

成果物確認は必要。
ただし、巨大なSTT全文、LLM全文、動画本体をUIや状態へ直接埋め込まない。

## AIエージェントAPIの考え方

```text
GET  /api/agent-requests/next
POST /api/agent-requests/:id/claim
POST /api/agent-requests/:id/complete
POST /api/agent-requests/:id/fail
```

意味:

- `next`: 実行可能な承認済み作業を1件返す。
- `claim`: AIエージェントがその作業を担当中にする。
- `complete`: 外部処理の結果と成果物参照を返す。
- `fail`: 失敗理由を返す。

`claim` が必要な理由:

- 複数runnerで同じ作業を二重実行しないため。
- 途中停止した作業を復旧対象として扱うため。
- 誰がいつ作業を引き受けたかをログに残すため。

## runnerの現在地

runnerは追加済み。
目的は、UI依頼からAPI実行完了までの経路を通しながら、外部処理はbackendではなくrunner側で行うこと。
現時点では、承認APIがrunnerをバックグラウンド起動し、runnerがAPIだけで作業取得と完了報告を行う。
承認APIはrunner完了まで待たず、UIは状態APIで進捗を確認する。

runnerの最低限の内訳:

- API client。
- 実行ループ。
- 作業種別ディスパッチ。
- 動画取り込み、STT、テーマ候補、構成、演出、動画生成のhandler。
- 成果物参照の返却。
- 失敗報告。

STTサーバ、ffmpeg、yt-dlpはrunnerの環境変数で差し替える。
Gemini APIの使用モデルはUIの依頼設定で用途から選び、runnerの標準モデルだけを変える場合は環境変数で差し替える。
Gemini APIの標準モデルは3.5だが、接続確認やJSON応答確認だけのテストでは2.5または3を明示して使う。
完成品レビュー用のWeb版Geminiはまだ接続していない。

## 最初から形を入れる作業種別

```text
prepare_video
run_stt
propose_clip_themes
build_clip_composition
create_edit_plan
apply_adjustment
render_video
```

`propose_clip_themes` の後で、人間が切り抜きたいテーマを選ぶ。
`build_clip_composition` は、選ばれたテーマに関係する複数箇所を集めて構成案を作る。
`create_edit_plan` は、構成案に含まれる複数の動画箇所をGemini APIへ渡し、テロップと画面の見せ方を含む演出案を作る。

## まだ作っていないもの

- API契約テスト。
- claimの排他制御。
- claim失敗時の復旧。
- 成果物本体の保存。
- 操作ログ。
- 認証。
- 停止、再実行、差し戻し、却下API。
- Web版Geminiによる完成品レビュー。

## 現時点で確認してほしいこと

1. UIで依頼を作り、「承認してAIに渡す」後にrunner完了まで進む流れ。
2. UIに残す情報量。現状は依頼、状態、成果物参照に絞っている。
3. 成果物確認の粒度。現状は成果物種別、参照URI、要約を最小単位にしている。
4. AIエージェントAPI。現状は `next -> claim -> complete/fail` で作業取得、担当中化、完了、失敗を分けている。
5. テーマ選択と複数箇所構成の流れ。候補決定が映像や音声の補助計測に戻っていないこと。

## レビュー後の着手候補

最優先候補は、runner経路をAPI契約テストで固定すること。
