# ZEV 開発方針整理 — 必要なこと・必要ないこと

## 人間向けサマリ(これだけ読めば判断できる)

**何を作るか**: 配信(VTuber)の切り抜きショート動画を自動生成・投稿するパイプライン。フェーズ1は大手事務所の許諾済み切り抜き市場で自分のチャンネルを運営し、「人間の切り抜き師の中央値に並ぶ品質」を速度と量つきで証明する。フェーズ2でその実績を持って、人間の切り抜き師が経済的に成立しない「薄い需要」(企業勢・中堅事務所・個人勢)に展開する。

**原理は3つだけ**。①LLMは意味判断のみ、数値・配置・境界はルールが決定的に決める。②同じ入力なら同じ出力にして、全判断のログを残す。③数値は全部仮置きでよく、公開後の実績で較正する。仕様の追加・削除に迷ったらこの3行に戻る。この原理に乗らない機能(LLMに自由にやらせる系、ログが残らない系)は足さない。

**今やるのは5つ**: 切り抜き境界のルール化 / 機械チェック(第1層バリデータ) / 演出4点セット / Gemini Web評価のAPI置換 / 決定ログ。このうち決定ログだけは後付けできないので、分析は将来でいいから記録は初日から入れる。

**作らないもの**が方針の半分を占める: 凝った演出、タイムライン編集UI、エージェントチャットUI、リアルタイム進捗表示、ダッシュボード、ジャンル自動判定、汎用プロファイル機構、マルチプラットフォーム、許諾管理(フェーズ2まで)、LLMスコア予測モデル。理由は各節に明記。

**人間が判断するのは3つだけ**: 山場の確認 / 公開承認 / 演出の濃さ(3択)。UIはこの判断を30秒で捌くためだけに存在する。

## 1. 戦略(フェーズ構成)

### 必要
- **フェーズ1**: 大手事務所(ホロライブ/にじさんじ等)の包括許諾に乗り、自分の切り抜きチャンネルを運営。目的はシェアではなく「人間の切り抜き師と同等品質」の達成と公開実績。競争軸は品質=中央値レベル、速度と量で圧倒。
- **フェーズ2**: 薄い需要(企業勢・中堅事務所・個人勢・VTuber外配信)に展開。AIの限界費用ゼロが武器。検証する変数は支払い意思のみ(品質はフェーズ1で証明済みの前提)。
- **移行判定**: 人間の切り抜き師の「中央値」に並んだら移行。トップに勝つ必要はない。判定の本番数字は月1のブラインドペア評価(勝率45〜55%で「同等」)。
- **勝負どころ**: 「配信終了 → 投稿可能」のリードタイム。目標60分以内、将来30分。
- 4〜6ヶ月で品質未達でも、速度・量を武器にフェーズ2の1社を試す並行ルートを持つ。

### 不要
- シェア獲得・トップ品質の追求(フェーズ1の目的ではない)。
- 1フェーズで複数変数を検証すること(フェーズ1=品質のみ、フェーズ2=支払い意思のみ)。

## 2. 核となる3原理(全領域に優先)

1. **LLMは意味判断、数値はルール**。LLMの出力は「演出指示」ではなく「意味アノテーション」(ここが山場/この語を強調/この文を使う)に留める。秒数・配置・頻度・座標はルール表とテンプレートが決定的に導出する。
2. **決定的にしてログを残す**。同じ入力→同じ出力。どの文を採用したか、どのルールでどの演出を入れたか、人間が何を直したかを構造化ログに残す。これがテスト・デバッグ・帰属分析(どの判断が成績に効いたか)の前提。**ログは後付けできない唯一の要素**。
3. **数値は仮置きして実績で直す**。閾値・スコア・重みは設定ファイルに出し、公開後の維持率データと突合して直す。完璧主義は不要。

## 3. パイプライン構造

### 必要
- 8ステップのパイプライン(v2.0最小構成)。冒頭2ステップ: [1]配信終了の検知(YouTube API/RSS監視→自動起動) [2]取り込みと事前解析(音声抽出→単語タイムスタンプ付きASR、音響3系列(音量・ピッチ・発話速度)の統計化。全部ローカル機械処理・LLMなし)。
  ※Keepのv2.0メモはStep[2]の途中で切れている。完全版はチャット側の project_summary v2.0 にあるはずなので、入手したらこの文書を更新する。
- **構築順序は依存関係で3段階**。Step 2完了=投稿開始。終了検知や承認UIの自動化(Step 3)は速度勝負用の装備であり、核心の問い「モーメント選定が人間に並ぶか」はStep 2時点で検証を始める。
- **検出器の並列化**: 意味検出(LLM)を検出器の1つ(D1)に格下げし、D2音響・D3映像・D4チャット流速と並列に走らせ、統一フォーマットの候補イベントストリーム {start, end, detector, intensity, evidence} に合流させる。VTuber文脈ではD2+D4だけで人間のモーメント発見をほぼ代替できる(チャット流速5倍超は強候補で、実装は単なる集計)。
- ジャンル差は**プロファイル(設定の束)**で吸収: 変わるのは抽出単位・価値関数・構成文法の3定義だけ。検出器は常に全部走らせ、プロファイルは「どれを信じるかの重み」。
- 実装するプロファイルは **vtuber_clip の1つだけ**(音響3型+チャット2型の最小限)。
- **文脈歪曲チェックは必須**(事務所ガイドラインは誤解を招く切り取りを禁止。違反は一発で許諾が飛ぶ)。
- チャンネル別**固定クロップ**(VTuber配信画面はレイアウトほぼ固定。タレントごとにプリセット)。

### 不要
- 自動化の先行作り込み(品質検証が後ろにずれる)。
- 顔トラッキング(アニメ顔は顔検出が効きにくい。固定クロップで代替)。
- ジャンル自動判定(最初は人間が承認時に分かる)。
- ジャンル別パイプライン(構造は共通、差分はプロファイルのみ)。
- talk_essence(意味駆動)プロファイルの実装(仕様書のまま保留)。

## 4. 切り抜き(モーメント選定と境界)

### 必要
- 「切り抜きポイントの特定」を**5つの問題に分離**する: ①トピック分割 ②採用判断(1話=1本の分割判断含む) ③内部圧縮(どの文を残すか) ④並べ替え(フック前出し) ⑤境界決定。①〜④が意味判断(LLM)、⑤は純粋にルール。
- VTuberでは「文の選択」が「モーメント検出」(音響+チャット)に置き換わるが、境界ルール(Stage F)は共通。
- 境界ルールの初期値: カット点は単語境界必須 / in点=文頭−120ms / out点=文末+150ms / 250ms以上の呼吸ポーズ優先 / 文中カット禁止 / 上昇イントネーション直後のout禁止 / 音声のみ20〜40msクロスフェード / ±50msをゼロクロス点にスナップ。
- 文アノテーションに **droppable / dependency(指示語の参照先)** を持たせ、圧縮時に「指示語が宙に浮いた切り抜き」を機械的に防ぐ。payoff文とそのdependencyチェーン、hook文は削除禁止。
- 最終検証: 機械検証(尺・文中カット・dependency整合)+ 採用文テキストのみのLLMチェック(「単体で意味が通るか」。映像不要・数円で切り抜き失敗の最終防衛線)。
- 構成文法はジャンルで逆転する: 意味系=結論前出し、情動系=トリガー提示→リアクション(リアクション先出しはネタバレで死ぬ)。情動系の文脈供給は状況テロップ1枚(「ホラゲー挑戦中」)で済ませる。

### 不要
- LLMに秒単位のin/out点を直接決めさせること(タイムスタンプ精度が出ない・「情報的に正しい」箇所を選びがち・長尺でブレる)。
- ジャンルの網羅的分類学を事前に完成させること(較正可能なアーキテクチャだけ先に作る)。

## 5. 演出

### 必要
- **4点セット+座布団だけ**: 字幕+キーワード強調 / パンチイン(1.0→1.15倍の瞬時切替) / SE3種(テロップ出現・強調・場面転換) / タイトルカード / 字幕の座布団(背景帯)。上位チャンネルも実際これしか使っていない。
- 5段階の決定ロジック: Stage1 意味アノテーション(LLM: importance/beat_type/emphasis_words/tone/pause_intent) → Stage2 ルール表で候補生成 → Stage3 制約つき貪欲選択(SE頻度上限・視覚変化間隔・密度予算) → Stage4 テンプレートで具体値(LLM関与ゼロ) → Stage5 検証。
- 採用基準は「**適用条件を機械的に書けるか**」。書けない演出(センスが要る演出)は出力がブレて品質の下振れ要因になるので入れない。
- VTuberでは「ピークを邪魔しない抑制」を加える: 絶叫にSEを被せない、溜めの無音を残す(pause_intent=trueで演出抑制の負のルール)。演出は足すより引く方が品質を決める。
- 無音・フィラーの自動カット(ジャンプカットは隠さずテンポ演出として使う)。

### 不要
- 凝ったトランジション(回転・3D・グリッチ。古く見える)/ モーショングラフィックス / 複雑なカラーグレーディング(固定LUT1つで足りる)/ アバター演出 / フォント多様化(1〜2書体固定の方が品質が安定)。
- LLMに配置・頻度・座標を決めさせること(「2〜4秒ごとに変化」をLLMは守れない)。
- 厳密最適化ソルバー(ILP等。貪欲+穴埋めで実用上差が出ない)。

## 6. 評価・品質保証

### 必要
- **3層スコアリング**:
  - 第1層: ルールベース検証(LLM不要・決定的・約35項目: 尺/無音残り/セーフゾーン/ラウドネス-14LUFS/静止3秒/SE頻度など)。pass/fail+違反リストで自動修正に差し戻し。品質バグの大半はここで止まる。
  - 第2層: LLM評価(API・固定ルーブリック7軸・JSONスキーマ強制・モデルバージョン固定・temperature 0〜0.3・ルーブリックをバージョン管理)。軸別スコア+理由+タイムスタンプ必須(総合点だけでは改善につながらない)。evidence(タイムライン引用)必須化で捏造を防ぐ。hook_strengthは単独足切り(2以下は他が満点でも差し戻し)。絶対評価よりペアワイズ比較を併用。再編集ループは上限2回。
  - 第3層: 公開実績による較正(LLMスコアと維持率カーブ等をペアで蓄積し、相関の弱い評価軸はルーブリック改訂か重み下げ)。**これをやらないと「LLMの好みに最適化した動画」を量産する**。
- フェーズ1は人間(自分)が毎本承認するので、**第2層は事故防止に絞った簡易版でよい**。
- 評価コスト設計: 第1層で足切り→第2層は構造化データ+サンプルフレームで安価に→フル映像評価は人間提示前の1回だけ。
- **品質比較の5層**(対人間): 層0=人間の切り抜き師の選択データセットを自動構築(人間クリップのASR→元配信と突合してin/out特定。Step 2直後に作る。閾値較正・境界改善・将来の教師データを兼ねる資産) / 層1=選定再現率(毎配信・自動。複数の切り抜き師が共通して選んだ区間の再現率95%以上が最低線) / 層2=境界偏差(自動)+ブラインドペア評価(月1・50ペア。これが移行判定)+LLMペアワイズ(毎日。必ず人手評価で較正。**代理であって代替ではない**) / 層3=正規化した公開成績(チャンネル内パーセンタイル・マッチド比較・自チャンネル維持率トレンド) / 層4=自己修正率(毎日・無料の日次シグナル)。

### 不要
- Gemini Web(ブラウザ)経由の評価(再現性なし・構造化出力なし・スケールしない・規約グレー)→ API+固定ルーブリックに置換。
- 品質を1つの数字で比べること / 再生数の生比較(チャンネル規模等の交絡で無意味)。
- LLMスコアの予測モデル化(データ数百本溜まってから)。
- ゴールデンセット自動リグレッション(当面は固定の検証用動画5本の目視で代替)。

## 7. データ

### 必要
- **3層構造**: 生データ(動画)/ 構造化イベント(発話・音響・映像・チャットのタイムライン)/ 知見(ルール・実績)。エージェントは基本イベント層で判断し、必要なときだけ特定区間をマルチモーダル確認する二段構え。フル文字起こしを毎回渡さない。
- 発話データの粒度: 単語レベルタイムスタンプ / フィラーフラグ / 役割ラベル(結論・根拠・脱線等。LLM前処理で付与)/ 感情・テンション値 / 無音区間 / 笑い声等の非発話イベント / チャット密度。
- **決定ログ+human_overrides を初日から**: 採否とその理由 / 適用演出と根拠ルールID / 人間の修正と却下理由。実績データ(維持率カーブ・離脱点)と突合して初めて「どの判断が効いたか」が分析できる。
- アセットカタログ(SE・BGM・テロップスタイルのメタデータ+適用条件)。
- 生ログ→知見への蒸留はバッチLLMで自動化(ナレッジベースへ還流)。

### 不要
- 巨大な解析結果・STT全文・LLM全文・動画本体を状態(正本)へ埋め込むこと(zev2固定方針と同じ。FileRef参照で持つ)。
- トレンドデータ・競合ベンチマーク・コメント分析(「あると差がつく」C級。フェーズ1の必須ではない)。

## 8. UI

### 必要(画面は3つだけ)
- **画面1 承認キュー(ホーム・ほぼ唯一の日常画面)**: カードに載せるのは人間が30秒で判断する材料のみ=検出根拠(なぜAIがここを選んだか)・歪曲チェック結果・テロップ案。warn付き候補は枠色で目立たせ、安全なものから捌ける順に並べる。「承認して投稿」で全自動。**却下時の理由1タップ(場面がつまらない/切り方が悪い/伝わらない/リスク)が較正ログになる——UIで唯一こだわる価値のある入力**。
- **画面2 候補の詳細**: 縦動画プレビュー+音声波形+検出イベントのタイムライン。境界修正は in/out点を文単位・±0.5秒でずらすナッジボタンだけ。**自由編集を許した瞬間に動画編集ソフトの劣化版になる**。ナッジで直らないクリップは却下して理由を残す(UIではなくパイプラインで直す問題)。
- **画面3 配信ビュー(週1の分析画面)**: 配信全体のタイムラインに自システムの検出モーメントと人間の切り抜き師の採用区間を重ね、選定再現率の答え合わせをする。開発用なので最初は雑でいい。
- 設計の軸: エージェントの成果物は「進捗」ではなく「**あなたの判断を待つ候補**」。この承認体験はフェーズ2の顧客向け画面とほぼ同構造(通知→候補を見る→承認)。
- 移行: 現zev2の「動画指定してキック」は配信終了検知ができるまでの手動トリガーとして残し、出口を進捗表示から承認キューに差し替える。

### 不要
- リアルタイム進捗表示(失敗時だけ通知すれば十分)/ タイムライン編集機能 / エージェントとのチャットUI(定型判断を捌く道具としてはボタンより遅い)/ ダッシュボードの数字群(計測指標は週次でスプレッドシートでいい)。
- 処理ログ・パイプライン状態・パラメータの画面表示。
- 個別演出(ズーム1個・SE1個)を承認するUI(直されたら理由をログに残すだけ)。
- AI向けUI・JSON閲覧画面の主導線化(zev2固定方針)。

## 9. 人間の意思決定ポイント(これだけに絞る)

1. **山場の確認**: 「山場は○○と判断した」← 違ったら差し替え。演出は自動で追従する。
2. **公開承認**: スコアと理由を見てGO/NG。
3. **演出の濃さ**: 控えめ/標準/濃いめ の3択(必要な時だけ)。

## 10. 実装規律(zev2リポジトリの固定方針)

- パッケージ管理はpnpm固定。Electronなし。共有型は packages/shared 優先。
- backend内で実STT・実LLM・実動画生成・実Gemini確認を実行しない(実処理はrunner/Codex側、backendは結果と成果物参照のみ受け取る)。
- 人間承認なしで動画生成まで進めない。後方互換の分岐を入れない。独自判断の係数や重み付けで挙動を決めない(明示された観点・ユーザーが決めた評価段階のみ)。
- 実処理接続の開始条件: API契約テスト・claim排他制御・成果物保存・操作ログ・セキュリティ境界(task-002/004/005/006/009)が揃ってから。
- 秘密情報・認証情報・個人情報を外部LLMへ送らない。APIキーをbackend/リポジトリに置かない。

## 12. メモ間の整合に関する注記
