# Web Geminiレビューから見えた編集機能候補

作成日: 2026-06-29

## 目的

完成動画をWeb Geminiに見せて演出レビューを受ける時、Geminiが実装可能な助言を返せるようにする。

このメモは実装指示ではない。Geminiとのやりとりで、編集機能として持つと効果が大きそうなものを記録する。

## 現在の前提

- Web Geminiレビューは完成動画だけを見る。
- レビュー対象は演出だけにする。
- テーマ、編集元場面、断片の順番、元動画、音声品質、投稿作業はレビュー対象外にする。
- レビュー結果を反映する時は、演出作成前から編集コピーを作ってやり直す。
- 現行の演出作成は、断片単位の画面パターン、検出範囲、テロップ文言、テロップの発話ID対応を扱う。

## 効果が大きそうな編集機能候補

### 1. テロップ配置候補の明示化

Geminiは、テロップが顔や手描き画面に重なっている点を強く指摘した。

現行でも話者の顔を避けた配置や、画面側への配置はあるが、レビューでは「白板の空き」「左右の余白」「顔の下ではなく別の空間」のように、より具体的な配置変更が出ている。

候補:

- テロップごとに配置候補を複数持つ。
- 候補は「画面側中央」「画面側上部」「話者顔を避けた下部」「話者顔を避けた上部」など、人間が読める名前にする。
- Geminiレビューには、どの候補なら問題が減るかを選ばせる。
- 数値座標をGeminiに直接決めさせない。候補名を選ばせ、座標計算はプログラム側で行う。

### 2. 発話IDに紐づくテロップ再分割

Geminiは、長いテロップを発話の意味に合わせて分ける提案を返した。

これは効果が大きい。テロップのタイミング問題は、時刻ではなく発話IDで扱う方針と相性が良い。

候補:

- Web Geminiレビューの改善指示を、発話IDベースのテロップ再分割案へ変換する工程を作る。
- 変換後は `sourceSpeechIds` と表示文で確認できるようにする。
- 人間にはIDを主表示しない。画面では「どの発話文に対応するテロップか」として見せる。

### 3. 同じ編集元場面内の演出用サブ断片分割

Geminiは、見せ場直前で画面パターンを切り替える提案を何度も返した。

現行の依頼文では「断片途中の画面パターン切り替え」は変更しないことにしている。ただし、見せ場の直前で構図を変える効果は大きそうに見える。

候補:

- 編集元場面そのものは変えず、同じ場面の中を発話ID境界でサブ断片に分けられるようにする。
- サブ断片ごとに画面パターンを変える。
- 切り替え位置は時刻ではなく発話ID境界で扱う。
- この機能を入れるまでは、Geminiには断片途中の画面パターン切り替えを提案させない。

### 4. 見せ場テロップのスタイル切り替え

Geminiは、見せ場に対して文字色、強調、表示タイミングの変更を提案した。

現在のテロップは見た目の品質が重要なので、毎回細かい数値を返させるのではなく、スタイル候補を選ばせる方がよい。

候補:

- テロップに「通常」「見せ場」「補足」「ツッコミ」のような役割を持たせる。
- 役割ごとにRemotion側のテロップ設定データを切り替える。
- Geminiには数値指定ではなく、役割や意図を返させる。
- 文字を小さくする方向の解決は避け、改行、分割、配置、スタイルで解決する。

### 5. レビュー反映前の人間選択

Geminiのレビューには有用な指摘と、現行機能では実装できない指摘が混ざる。

そのまま全部反映すると、人間のコントロールを外れる。

候補:

- Web Geminiレビューを「採用する指摘」「今回は採用しない指摘」に分ける画面を作る。
- 採用した指摘だけを演出作成前からのやり直し理由に入れる。
- 採用しない指摘もログには残す。
- 自動採用はしない。

## Geminiに伝えるべきこと

レビュー依頼文には、少なくとも次を入れる。

- 変更できること。
- 変更しないこと。
- テロップを小さくして解決しないこと。
- 時刻ではなく、最終的には発話IDに紐づけて反映すること。
- 数値座標や根拠のない係数ではなく、人間が読める変更意図を返すこと。

## 現時点の判断

最優先候補は、テロップ配置候補と発話IDベースのテロップ再分割。

次に、同じ編集元場面内のサブ断片分割を検討する価値がある。

この3つがあると、Web Geminiのレビューを演出作成へ返す流れがかなり実用に近づく。
