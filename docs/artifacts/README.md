# zev2 成果物仕様

このディレクトリは、7工程の間で渡す成果物の人間向け仕様です。
目的は、コードを読まなくても「何が入力で、何が成果物として次へ渡るか」を確認できるようにすることです。

## 読み方

- `入力`: その成果物を作る時に読む前工程の成果物。
- `作る工程`: runner が実行する工程。
- `次に読む工程`: この成果物を入力として使う工程。
- `最低条件`: 実行時検証で落とすべき条件。

## 成果物一覧

| 成果物 | 作る工程 | 次に読む工程 | 意味 |
| --- | --- | --- | --- |
| 入力動画参照 | 動画取り込み | STT、演出作成、動画生成 | 後続工程が読める動画の場所 |
| 文字起こし | STT | 内容候補整理、使用素材構成 | 発話ID、開始、終了、話者、本文 |
| 内容候補 | 内容候補整理 | 内容選択、使用素材構成 | 文字起こし内にある出来事、話題、反応の説明 |
| 使用素材構成案 | 使用素材構成 | 使用素材確認、演出作成 | 選ばれた内容に関係する発話箇所と動画としての流れ |
| 編集案 | 演出作成 | 微調整、動画生成 | 画面枠、テロップ、動画断片の見せ方 |
| 調整結果 | 微調整 | 動画生成 | 編集案を動画生成へ渡すための確認結果 |
| 完成動画 | 動画生成 | 生成後レビュー | 音声付きMP4 |
| 工程manifest | 各工程 | 人間の確認、失敗時の切り分け | 実際に読んだ成果物と作った成果物の記録 |

## 文字起こし

入力:

- 入力動画参照。

作る工程:

- STT。

次に読む工程:

- 内容候補整理。
- 使用素材構成。

最低条件:

- `kind` は `transcript_json`。
- `mode` は `zev-local-stt` または `zev-sample-stt`。
- 発話は1件以上ある。
- 各発話は、整数の発話ID、開始時刻、終了時刻、本文を持つ。
- 終了時刻は開始時刻より後。
- 発話まとまりは、存在する発話IDだけを参照する。

サンプル:

- [transcript.json](samples/transcript.json)

## 内容候補

入力:

- 文字起こし。

作る工程:

- 内容候補整理。

次に読む工程:

- 内容選択。
- 使用素材構成。

最低条件:

- `kind` は `theme_json`。
- `mode` は `sample-theme-options` または `transcript-content-options`。
- 内容候補は1件以上ある。
- 各候補は、ID、タイトル、概要、代表発話ID、関連発話IDを持つ。
- 代表発話IDと関連発話IDは、文字起こしに存在する発話を指す。
- 概要は、AIが面白いと断定する文ではなく、文字起こし内に何があるかを説明する文にする。
- 構成メモは、人間がこの内容で作りたいか判断するための短い説明にする。

サンプル:

- [themes.json](samples/themes.json)

## 使用素材構成案

入力:

- 文字起こし。
- 内容候補。
- ユーザーが選んだ内容。

作る工程:

- 使用素材構成。

次に読む工程:

- 使用素材確認。
- 演出作成。

最低条件:

- `kind` は `composition_json`。
- 選ばれた内容候補IDを持つ。
- 使う場面は1件以上ある。
- 各場面は、開始時刻、終了時刻、役割、発話ID、発話本文、つなぐ理由を持つ。
- 各場面の発話IDは、場面内の発話単位と一致する。
- 人間が細かい発話IDや秒数を選ばなくても、この素材で動画にできそうか判断できる説明を持つ。

サンプル:

- [clip-composition.json](samples/clip-composition.json)

## 編集案

入力:

- 使用素材構成案。
- 必要な動画断片。

作る工程:

- 演出作成。

次に読む工程:

- 微調整。
- 動画生成。

最低条件:

- `kind` は `edit_plan_json`。
- `mode` は `gemini-api-edit-plan` または `sample-edit-plan`。
- 動画断片は1件以上ある。
- 各動画断片は、開始時刻、終了時刻、役割、発話ID、発話単位、画面枠を持つ。
- テロップ案は1件以上ある。
- テロップ案は、表示文と対応する発話IDを持つ。
- テロップ案の発話IDは、編集案内の動画断片に存在する発話IDだけを指す。

サンプル:

- [edit-plan.json](samples/edit-plan.json)

## 調整結果

入力:

- 編集案。

作る工程:

- 微調整。

次に読む工程:

- 動画生成。

最低条件:

- `kind` は `patch_json`。
- 編集案の参照を持つ。
- 生成可能かどうかを持つ。
- 変更内容は、人間が読める対象、処理、理由で書く。

サンプル:

- [adjustment-patch.json](samples/adjustment-patch.json)

## 固定確認と通常実行

固定確認を使う場合:

- `config/runtime.jsonc` で対象工程の `mode` を `fixed` にする。
- 文字起こし、内容候補、編集案は `sample` 系の `mode` として保存する。

通常実行の場合:

- 固定データへ逃げない。
- STT接続やGemini API接続がなければ、その工程を失敗にする。

## 守ること

- 1つの成果物に複数工程の判断を混ぜない。
- 後続工程は、前工程の内部状態ではなく成果物を読む。
- UI表示用の一時状態を工程間の正本にしない。
- AI応答は、必ず成果物構造へ変換してから次工程へ渡す。

## 工程manifest

工程manifestは、各工程の実行後に成果物ディレクトリへ保存する短いJSONです。
状態ファイルには入れません。

保存名:

- `{工程名}-manifest.json`

目的:

- その工程が実際に読んだ成果物参照を確認する。
- その工程が実際に作った成果物参照を確認する。
- 固定確認、API実行、動画生成のどこで失敗したかを切り分ける。

最低条件:

- `kind` は `workflow_step_manifest`。
- 工程ID、依頼ID、編集コピーIDを持つ。
- 入力成果物参照を持つ。入力がない工程は空配列でよい。
- 出力成果物参照を持つ。
- 成果物の `mode` がある場合はmanifestにも残す。

入力として残すもの:

| 工程 | manifestに残す入力 |
| --- | --- |
| 動画取り込み | なし |
| STT | 入力動画参照 |
| 内容候補整理 | 文字起こし |
| 使用素材構成 | 文字起こし、内容候補 |
| 演出作成 | 使用素材構成案、入力動画参照 |
| 微調整 | 編集案 |
| 動画生成 | 編集案、入力動画参照 |
