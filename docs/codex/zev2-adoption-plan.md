# ZEV2 Fable案適用準備計画

作成日: 2026-06-13
管理主体: Codex

## 位置付け

この文書は、Fable 5 が作成した `docs/claude/development-policy.md` を有用な草案として読み、ZEV2 の現行実装を調査した上で、今後の適用方針を Codex 管理の作業基準として整理する。

`docs/claude/development-policy.md` は正典ではない。ChatGPT との会話内容も正典ではない。今後の作業基準は、Codex がリポジトリ内に記録し、ユーザーが確認した内容とする。

## 本来の目的確認

この準備タスクの目的は、動画品質改善や演出実装ではない。Fable案をZEV2へ適用する前に、AIエージェントの判断がブラックボックスにならず、ユーザーが状態遷移、判断理由、参照データ、人間承認の必要箇所を確認できる制御層を設計することである。

## リポジトリ調査から見えた現在地

現行ZEV2は、動画処理そのものではなく、次の骨格を先に通す実装になっている。

- 人間が実行前下書きを作成する。
- 人間が下書きを承認する。
- 承認後にAIエージェント用の作業キューが作られる。
- dry-run runner がバックエンドAPIだけを使って作業を取得し、取得済みにし、完了または失敗を報告する。
- 工程ごとの成果物は本体ではなく参照として保存される。
- UIは依頼、処理状態、成果物参照、履歴を最小表示する。

調査した主な実装は次の通り。

| 領域 | 現行実装の意味 | 関連ファイル |
| --- | --- | --- |
| 共有型と工程定義 | 動画取り込み、STT、候補探索、Gemini候補確認、演出付与、微調整、動画生成の7工程を定義している。 | `packages/shared/src/index.ts` |
| 実行前下書き | 人間の依頼入力と承認状態を保持している。 | `packages/shared/src/index.ts`, `backend/src/routes/control.ts` |
| 承認後の作業キュー | 承認済み下書きから直列依存のAI作業を作る。 | `packages/shared/src/index.ts`, `backend/src/routes/control.ts` |
| 状態遷移 | 作業待ち、前工程待ち、実行中、完了、失敗の最小状態を扱う。 | `packages/shared/src/index.ts`, `backend/src/routes/control.ts` |
| AIエージェント入口 | 次の作業取得、取得済み化、完了報告、失敗報告のAPIを提供している。 | `backend/src/routes/control.ts` |
| dry-run runner | 実処理なしで全工程をAPI経由で完了させる。 | `runner/src/index.ts`, `backend/src/runner/auto-runner.ts` |
| 状態保存 | 開発用の実行状態を `runtime/state.json` に保存する。 | `backend/src/store/json-store.ts` |
| 成果物参照 | 工程成果物の本体ではなく、参照URI、種類、MIME typeを保存する。 | `packages/shared/src/index.ts`, `backend/src/routes/control.ts` |
| UI表示 | 実行中工程、成功工程、失敗工程、履歴、成果物件数を表示する。 | `client/src/App.vue`, `client/src/stores/controlQueue.ts` |

現行実装は「AIエージェントがいくつかのプロセスを経て終了状態に到達する」骨格としては成立している。一方で、Fable案をそのまま入れるには、次の制御が足りない。

- AIエージェントがなぜ候補を選んだかを構造化して保存する仕組みがない。
- 判断、根拠、参照データ、提案する次状態、人間に求める判断が、成果物参照や完了理由と分離されていない。
- 工程状態は最小の実行状態であり、レビュー待ち、承認待ち、投稿可能、公開済み、最終完了のような重要状態は未定義。
- 人間承認は実行前下書きの承認に限られており、候補採用、編集案確定、動画生成、投稿可能化、最終完了の各段階を止めるゲートにはなっていない。
- 動画生成工程は人間承認が必要な工程として定義されているが、現在のdry-runでは最初の承認時に動画生成まで含めると、後続で別の承認を挟まずキューへ入る。
- ログ予定はAPI操作履歴の粒度であり、Fable案が求める判断ログの粒度には足りない。
- UIは進捗表示が中心であり、ユーザーがAI判断を確認して承認・却下・差し戻しする画面にはまだなっていない。

## 全体方針

Fable案の適用は、動画品質改善から始めない。最初に作るべきものは control plane、つまりAIエージェントの制御層である。

control plane は、AIエージェントが処理を進めるための「裏側の管理機能」ではなく、人間がAIエージェントの判断を追跡し、止め、承認し、差し戻せるようにするための正本である。

初期の目的は次の通り。

- 状態遷移を明示する。
- AIエージェントの判断を構造化ログとして残す。
- 人間承認なしに重要状態へ進めない。
- エージェント出力を自由文だけにしない。
- 判断、根拠、参照データ、次状態、人間に求める判断を記録する。

## 最初に適用すべき範囲

最初に適用すべき範囲は、現行の7工程を置き換えることではない。現行の実行経路を維持しながら、次の制御を先に設計する。

1. 工程状態と重要状態の分離
   - 実行待ち、実行中、完了、失敗は作業処理の状態として残す。
   - レビュー待ち、承認待ち、差し戻し待ち、投稿可能、公開済み、最終完了は人間制御の状態として別に扱う。

2. 判断ログの正本化
   - AIエージェントが作業を完了したことだけでなく、何を判断し、なぜそう判断し、どの成果物参照を根拠にし、次にどの状態を提案し、人間に何を確認してほしいかを保存する。

3. 人間承認ゲート
   - 候補採用、編集案確定、動画生成、投稿可能化、公開、最終完了に進む前に、人間の承認、却下、差し戻しを保存する。
   - 初期実装では投稿や公開そのものは作らないが、投稿可能状態や最終完了状態へ人間承認なしに進めない設計を先に固定する。

4. エージェント出力スキーマ
   - 完了報告の自由文に判断を混ぜない。
   - 候補、レビュー、編集案、確認要求を、意味が追える構造として保存する。

5. 最小UI表示
   - 最初のUIはJSON閲覧画面ではなく、AI判断、人間確認待ち、承認・却下・差し戻しの最小表示にする。
   - 候補詳細、演出編集、ダッシュボード、自由タイムライン編集は含めない。

## Fable案から採用する方針

- AIエージェントの判断は後から追える形で保存する。
- 成果物本体、巨大な解析結果、STT全文、LLM全文、動画本体を状態の正本に埋め込まない。
- ファイル参照で動画、書き起こし、候補、編集案、生成物をつなぐ。
- LLMは意味判断に使い、秒数、座標、演出頻度、投稿可否の自由決定には使わない。
- 数値、境界、配置、頻度は、ユーザーが確認した明示ルールで扱う。
- 人間が判断する箇所を絞る。
- 却下や差し戻しの理由を将来の改善材料として残す。
- backend はSTT、LLM、動画生成、Gemini確認を直接実行せず、AIエージェントやCodex側の実処理から結果参照を受け取る。
- 実処理接続の前に、API契約、排他制御、成果物保存、ログ、セキュリティ境界を固める。

## Fable案から修正して採用する方針

- Fable案では決定ログ、境界ルール、品質検証、演出4点セットが同じ初期候補として並ぶが、ZEV2では control plane を先にする。判断が保存されない状態で動画品質改善に進むと、改善理由を追えないためである。
- Fable案の「閾値・スコア・重みを設定に出す」考え方は、そのまま採用しない。ZEV2では独自判断の係数や重み付けを使わず、ユーザーが確認した評価段階と明示ルールだけを扱う。
- Fable案の「同じ入力なら同じ出力」は目標として扱うが、外部AIや外部動画確認が絡む箇所は、入力、モデルまたは確認手段、参照データ、判断理由を記録することで再確認可能性を先に確保する。
- Fable案のUI3画面案は将来候補として扱う。初期は承認キュー全体を作り込まず、現在の進捗表示に「判断ログ」と「人間確認待ち」を最小表示する設計から始める。
- LLMは固定フローで毎回呼ぶのではなく、エージェントAIが必要性を判断する。ただし、呼ぶ必要があると判断した理由、使った入力、出力の意味、人間確認が必要な点を必ず記録する。
- 人間中央値品質は最終目標ではなく最低合格ラインとして扱う。初期control planeでは品質比較の実装をしないが、将来の品質ゲートはこの前提を外さない。

## Fable案から保留する方針

- 演出4点セット。
- D4チャット。
- D4-lite。
- 自動投稿。
- 評価スコアリング。
- ダッシュボード。
- 配信全体タイムラインの分析画面。
- 人間クリップとの選定再現率計測。
- ブラインドペア評価の運用。
- 公開実績による較正。
- チャンネル別固定クロップ。
- 音響、映像、チャット検出器の並列化。

これらは重要な候補ではあるが、control plane がないまま入れると、AIの判断と人間の承認が追えないまま機能だけが増える。

## 現時点で実装しない方針

- 動画生成品質の改善。
- 実STT、実LLM、実Gemini確認、実動画生成の接続。
- 投稿または公開処理。
- 自動採用または自動足切り。
- 係数、重み、総合点による自動判断。
- タイムライン自由編集UI。
- AI向けUI。
- JSON閲覧を主導線にするUI。
- 既存ZEVとの後方互換分岐。

## 初期実装に含めないもの

- 演出4点セット。
- D4チャット。
- D4-lite。
- 自動投稿。
- 評価スコアリング。
- ダッシュボード。
- タイムライン自由編集UI。
- 動画生成品質の改善。

## 今後の実装順序案

| 順序 | 作業 | 処理の意味 |
| --- | --- | --- |
| 1 | control plane | AIエージェントの判断、状態、人間承認を扱う制御層を正本化する。 |
| 2 | 状態遷移 | 作業処理状態と人間制御状態を分け、許可される遷移を表で固定する。 |
| 3 | 決定ログ | AIの判断、理由、参照データ、提案する次状態、人間への確認要求を保存する。 |
| 4 | 人間承認ゲート | 重要状態に進む前に承認、却下、差し戻しを保存し、未承認の前進を拒否する。 |
| 5 | エージェント出力スキーマ | 自由文の完了報告とは別に、判断結果を構造化して返す契約を作る。 |
| 6 | 最小UI表示 | 判断ログと確認待ちを、人間が読める最小表示として出す。 |
| 7 | 候補生成の見える化 | 候補区間、根拠、参照データ、懸念点を確認できるようにする。 |
| 8 | 境界ルール | 秒数、境界、配置、頻度などをAIの自由裁量にせず、明示ルールとして扱う。 |
| 9 | 品質ゲート | 人間が確認した評価段階と明示ルールで品質確認を設計する。 |
| 10 | 演出 | control plane と品質確認が成立した後に、演出実装を開始する。 |

## 次に作るべき設計文書

次にCodexが提案すべき作業は、実装ではなく `docs/codex/control-plane-spec.md` の作成である。

その文書では、少なくとも次を固定する。

- 作業処理状態と人間制御状態の一覧。
- 許可される状態遷移表。
- 重要状態へ進む前に必要な人間判断。
- AI判断ログの保存形式。
- 人間の承認、却下、差し戻しの保存形式。
- エージェント出力に必ず含める判断項目。
- UIに最初に表示する最小情報。
- 実処理接続に進む前の確認手順。
