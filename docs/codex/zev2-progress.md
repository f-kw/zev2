# ZEV2 Codex進捗記録

作成日: 2026-06-13
管理主体: Codex

## 現在フェーズ

承認ゲートと型付き実行命令の骨格を作るために、control plane 最小実装と、工程ごとに評価できるUI再構成の最小実装を確認したフェーズ。

`docs/codex/control-plane-spec.md` の基本方針は、ユーザーから特に反対なしとして仮採用された。現在は、候補生成後と動画生成前でAIエージェントを人間確認待ちに止め、判断ログ、人間確認項目、人間の承認操作を保存し、UIから承認、却下、修正依頼を選べる最小実装が入っている。

次の目的は、ユーザーがUI上で流れと評価ポイントを確認できる状態を保ったまま、ZEVの既存実装を参考にしながら、zev2の責務分離を崩さずに候補生成、Gemini確認、編集案、動画生成の実処理接続へ進むことである。

## ユーザーレビューで判明した問題

### ZC-R-001: 承認画面で、何を見てどう判断すればよいか分からない

確認日: 2026-06-14

見えている問題:

- 現在の承認画面は、AIの判断、理由、根拠、次に起きる処理を表示しているが、それを人間がどう読んで承認、修正依頼、却下を選ぶべきかが分からない。
- 候補生成後の確認と動画生成前の確認は、承認の意味が違う。前者は「候補を映像確認へ進めてよいか」、後者は「編集案を動画生成へ渡してよいか」である。しかし画面上では、その違いが十分に伝わっていない。
- 成果物は表示されているが、承認時にどの成果物を優先して見るべきかが分からない。
- 「承認して続行」を押すと後続で何が動くかは表示されているが、ユーザーの判断責任として何を確認したことになるのかが曖昧である。

人間にとっての影響:

- ユーザーがAIの判断を制御している感覚を持てない。
- 承認、修正依頼、却下の使い分けができない。
- control plane の目的である「AI判断をブラックボックスにしない」状態にまだ届いていない。
- 画面レビュー中に問題点が会話だけで流れると、後続実装で何を直すべきか追えなくなる。

次に必要な設計:

- 承認ゲートごとに、人間が見るべき判断ポイントを定義する。
- 候補生成後の確認では、対象動画、依頼内容、候補区間、候補理由、次に進める価値を確認対象にする。
- 動画生成前の確認では、選ばれた候補、編集案、開始位置、終了位置、テロップ案、生成して確認する価値を確認対象にする。
- 各承認ゲートで、承認、修正依頼、却下を選ぶ条件を人間向けに表示する。
- 承認後に開始される後続処理を、処理名だけではなく「何が起きるか」として表示する。

done 条件:

- 候補生成後の確認画面で、ユーザーが何を確認したら承認できるか分かる。
- 動画生成前の確認画面で、ユーザーが何を確認したら承認できるか分かる。
- それぞれの確認画面で、承認、修正依頼、却下の使い分けが画面上で説明されている。
- 承認時に優先して見る成果物が画面上で分かる。
- 承認後に開始される後続処理が、人間向けの意味で表示されている。
- ユーザーがJSONを読まなくても、承認判断の意味を説明できる。

### ZC-R-003: 承認画面で主な問いと判断材料が同じレベルに見える

確認日: 2026-06-14

見えている問題:

- 画面には依頼、実行状態、次の操作、人間確認、AIの判断、理由、次に起きる処理、根拠、人間の判断、処理の流れが並んでいる。
- しかし、ユーザーが最初に答えるべき主な問いが見えない。
- 現在の表示では、「この編集案で動画生成へ進めてよいか」という問いと、その判断材料である理由、根拠、成果物、処理状態が同じ重さで表示されている。
- `review_required` のような内部状態が、人間向けの問いより目立つ位置に出ている。
- 承認済みになった後も、何を承認した結果なのかが主語として残らず、「承認として記録」という処理結果だけが見えている。

人間にとっての影響:

- ユーザーは、承認欄を見ても「今、自分は何に答えるべきだったのか」を把握しづらい。
- 承認済みの結果を見返しても、「どの問いに対して承認したのか」が分かりづらい。
- 判断材料が並んでいても、どれが主判断で、どれが補助情報か分からない。

次に必要な設計:

- 承認画面は、最上位に主な問いを1つ表示する。
- 主な問いの下に、判断するためのサブ情報を置く。
- サブ情報は、確認対象、AIの提案、理由、根拠、承認後に起きる処理、人間の過去判断に分ける。
- 内部状態名は主表示にしない。必要なら補助情報として扱い、人間向けの意味に変換して表示する。
- 承認済みの場合も、「どの問いに対して、どの判断をしたか」を残す。

done 条件:

- 承認画面の先頭に、ユーザーが答えるべき問いが1つ表示されている。
- 候補生成後の確認では、「この候補を映像確認へ進めてよいか」が主な問いとして読める。
- 動画生成前の確認では、「この編集案で動画生成へ進めてよいか」が主な問いとして読める。
- AIの判断、理由、根拠、成果物、処理状態は、主な問いを判断するためのサブ情報として見える。
- 内部状態名が、主な問いや次の操作より強く見えない。
- 承認済みの確認項目でも、どの問いに対して承認したのかが分かる。

### ZC-R-004: 依頼入力欄が常に表示されていて、実行結果の確認を邪魔している

確認日: 2026-06-14

見えている問題:

- 依頼入力欄が画面右側に常時表示されている。
- 実行済みの結果や人間確認を見ている場面でも、新しい依頼を作るフォームが同じ強さで表示されている。
- ユーザーが今やりたいことは、作成済みの実行結果を見て、問題点や次の差し替え箇所を確認することである。
- 常時表示の入力欄により、画面の主導線が「今の結果を確認する」ではなく「次の依頼を入力する」に見えやすい。

人間にとっての影響:

- レビュー中に、どこを見ればよいか分かりづらくなる。
- 承認、確認材料、生成動画確認に必要な表示領域が狭くなる。
- 実行結果確認と新規依頼作成が同じ画面重みになり、現在の作業目的がぼける。

次に必要な設計:

- 実行結果を見ている間は、依頼入力欄を常時表示しない。
- 新規依頼作成は、必要なときに開く操作にする。
- 実行中、確認待ち、完了後レビューの画面では、現在の実行結果、主な問い、成果物、次に確認するものを優先表示する。
- 依頼入力は、未開始状態または「新しい依頼を作る」操作を選んだ時だけ主表示にする。

done 条件:

- 実行結果の確認中に、依頼入力欄が常時表示されない。
- 新規依頼を作る操作は画面上に残っている。
- 実行中、確認待ち、完了後レビューでは、現在の結果と承認判断が画面の主導線になっている。
- 依頼入力欄を閉じても、現在の実行結果、成果物、承認履歴を確認できる。

### ZC-R-005: UIテキストが全体的に開発者目線になっている

確認日: 2026-06-14

見えている問題:

- 画面内の文言が、利用者が何をすればよいかではなく、システム内部の役割や工程を説明している。
- 「AIエージェント」「API」「工程」「成果物」「人間確認」「微調整」「判断ログ」「review_required」のような開発者向けの概念が、利用者向けの主表示として出ている。
- 利用者が知りたいのは、内部でどの役割が動いたかではなく、今どの動画案を見ているのか、何を確認すればよいのか、押すと何が起きるのかである。
- 現在の文言は、実装構造を説明するには役立つが、操作画面としては判断や行動に直結しない。

人間にとっての影響:

- 利用者が、画面の意味を開発者用語に変換しながら読まなければならない。
- 承認、修正依頼、却下、成果物確認の判断が遅くなる。
- 実装の状態は伝わっても、利用者にとっての次の行動が伝わらない。
- control plane が「AIを制御する画面」ではなく「内部ログを見る画面」に見える。

次に必要な設計:

- UIテキストを、開発者向けの役割名ではなく利用者の行動、判断、結果の意味で書き直す。
- 工程名を出す場合も、それだけを主表示にせず、「何を確認する工程か」を併記または置き換える。
- 内部状態名、API名、保存形式名、処理キュー名は主表示にしない。
- 利用者に必要な文言は、問い、確認対象、判断条件、押した後に起きること、見返す結果に分ける。
- 開発者向けの詳細は、必要なら折りたたみ、補助情報、詳細表示に下げる。

done 条件:

- UI上の主要テキストが、利用者の行動または判断の意味で書かれている。
- 「API」「キュー」「内部状態」「判断ログ」のような開発者向け語が、主導線の文言として出ていない。
- 候補生成、編集案、動画生成などの工程が、利用者にとって何を確認する場面か分かる文言になっている。
- ボタン文言が、内部処理ではなく利用者が許可する内容を表している。
- 完了後の画面で、利用者が次に見るべきものと話すべき論点が分かる。
- 開発者向けの詳細情報は、主表示ではなく必要時に見られる補助情報になっている。

### ZC-R-006: JSON表示を開いた後に閉じられず、項目の読み方も分からない

確認日: 2026-06-14

見えている問題:

- 成果物のJSONは「中身を見る」で展開できるが、展開後に閉じる操作がない。
- JSONの内容は開発者向けの構造そのままで、利用者が各項目をどう読めばよいか分からない。
- JSONを承認判断の補助情報として使うには、項目の意味を同時に参照できる必要がある。
- 現在の表示では、JSON本文だけが出るため、値が何を意味し、承認判断のどこに関係するかが分からない。

人間にとっての影響:

- 一度JSONを開くと画面が長くなり、主な問いや生成動画確認へ戻りづらい。
- JSONの項目名を見ても、候補、区間、理由、テロップ、動画生成範囲などの意味に変換できない。
- JSONが判断材料ではなく、読めない内部データに見える。

次に必要な設計:

- JSONを展開した後に閉じる操作を付ける。
- JSON本文の横、または同じ表示領域内に、主要項目の意味を利用者向けに表示する。
- 項目の意味は、開いている成果物の種類ごとに変える。
- 候補JSONでは、候補ID、開始位置、終了位置、タイトル、冒頭文、候補理由、根拠発話の意味を説明する。
- 編集案JSONでは、選ばれた候補、動画に使う区間、冒頭文、テロップ案、各区間の役割を説明する。
- 微調整JSONでは、動画生成前に確定した変更点、変更理由、生成可能かどうかを説明する。
- 動画生成計画JSONでは、実素材を切り出したのか、確認用映像にしたのか、使った区間、出力サイズを説明する。

done 条件:

- JSONを展開した状態で、同じ操作位置から閉じられる。
- JSON表示中に、主要項目の意味を同時に確認できる。
- 項目説明は、開いている成果物の種類に合わせて変わる。
- 利用者がJSONの項目名を知らなくても、候補区間、編集案、生成対象、変更理由の意味を把握できる。
- JSON表示が、主な問いや生成動画確認より強く画面を占有し続けない。

### ZC-R-007: 処理の流れ図は有用だが、画面構成で活用できていない

確認日: 2026-06-14

見えている問題:

- 処理の流れ図自体は、全体像を伝える要素として有用である。
- しかし現在は、流れ図が画面内の一部要素として埋もれており、ユーザーが今どの段階を評価しているかを理解する入口になっていない。
- 1画面に依頼、状態、承認、成果物、JSON、流れ図、入力欄が同時に出すぎている。
- その結果、流れ図が「全体の地図」ではなく、補助的な進捗表示に見えている。

人間にとっての影響:

- ユーザーは現在位置を掴めないまま、承認欄や成果物を読むことになる。
- 各工程で何を評価すればよいか分からない。
- 前の工程へ戻って確認する導線が弱く、どの情報がどの段階に属するか分からない。

次に必要な設計:

- 処理の流れ図を画面上部に置き、全体の現在位置を示す主導線にする。
- 流れ図の下に、プロセスごとのタブまたは同等の切り替えUIを置く。
- 各タブでは、そのプロセスで必要な情報だけを表示する。
- ユーザーは現在の工程だけでなく、完了済みの工程にも戻って確認できる。
- 1画面にすべてを並べず、情報を工程ごとに分ける。

候補となる画面構成:

- 上部: 処理の流れ図。依頼、候補確認、動画生成前確認、生成動画確認、修正点整理などの現在位置を示す。
- 下部: プロセス別タブ。選択中の段階に必要な主な問い、確認対象、成果物、判断操作だけを表示する。
- 補助情報: JSON、内部状態、詳細な根拠は、各タブ内の折りたたみまたは補助欄で表示する。

done 条件:

- 処理の流れ図が画面上部にあり、現在位置が分かる。
- 流れ図の下で、プロセスごとに表示内容を切り替えられる。
- 完了済みの工程へ戻って確認できる。
- 各プロセスの画面には、その段階で評価する情報だけが主表示される。
- 依頼入力、JSON、内部状態、詳細ログが全プロセスで常時表示されない。

### ZC-R-008: 新しい依頼作成と現在の実行レビューが混ざっている

確認日: 2026-06-14

見えている問題:

- 「新しい依頼を作る」が依頼プロセス内にあり、現在の実行に属する操作のように見える。
- 新しい依頼作成は、現在の実行結果を確認する流れとは別の開始操作である。
- 「新しい依頼を作る」を押した後も、前の候補、動画生成前確認、生成動画確認、修正点整理が残ると、入力中の依頼と前回の結果が混ざる。
- 新しい依頼作成は、全体の流れより上に置き、押したら前回レビュー表示を隠す必要がある。

人間にとっての影響:

- ユーザーが、今見ている情報が新しい依頼の入力なのか、前回の実行結果なのか分からなくなる。
- 依頼プロセスの中に新規作成があると、現在の依頼内容の確認と別実行の開始が同じ意味に見える。

done 条件:

- 新しい依頼作成は全体の流れより上に表示されている。
- 依頼プロセス内には、現在の依頼内容の確認だけが表示されている。
- 新しい依頼作成を開いたら、前回の全体の流れとレビュー内容が隠れる。
- 入力を閉じると、前回の実行レビューへ戻れる。

### ZC-R-010: 候補確認で、誰の判断か、何を見て判断するかが分からない

確認日: 2026-06-14

見えている問題:

- 「保存済みの判断」がAIの判断なのか、人間の判断なのか、システム状態なのか分からない。
- 「提案」と「保存された判断」が同列に見え、AIの提案と人間の判断の順序が伝わらない。
- 「進める」「修正する」「止める」が、候補に対する操作なのか、実行全体に対する操作なのか分からない。
- 候補区間、候補理由、見どころ、懸念点、確認すべき箇所が主表示になっていない。

## 見つかった現行実装の概要

### タスク状態の定義

現行のAI作業状態は、作業待ち、前工程待ち、実行中、完了、失敗の5種類である。実行前下書きは、下書き、承認済み、却下の3種類である。

これらは処理実行の状態としては最小で成立しているが、候補レビュー待ち、編集案承認待ち、投稿可能、公開済み、最終完了のような人間制御の重要状態はまだ定義されていない。

### 状態遷移の実装

承認済み下書きから直列依存のAI作業が作られる。次の作業取得では、前工程が完了している作業だけが返る。取得済み化すると実行中になり、完了報告で完了になり、失敗報告で失敗になる。

現行の遷移は工程完了の順序を守るためのものであり、AI判断の確認、人間承認、差し戻し、却下、投稿可能化、最終完了を制御するための遷移ではない。

### AIエージェント処理の入口

AIエージェントは、次の作業取得、取得済み化、完了報告、失敗報告のAPIを使う。UIを自動操作して進める構造ではない。

### エージェントの終了条件

dry-run runner は、次の作業取得で作業が返らなくなると終了する。最大処理件数に到達した場合も停止する。

この終了条件は「実行できる作業がない」ことを示すだけで、ユーザー承認済みの最終完了や投稿可能状態を示すものではない。

### 成果物保存の仕組み

現行では成果物本体を状態に埋め込まず、成果物参照を保存する。参照には種類、URI、MIME type、アクセス種別、作成時刻が含まれる。

成果物本体の保存領域、参照検証、欠落時の扱い、後工程の取得方法は後続タスクとして残っている。

### ログの仕組み

現在は詳細な操作ログや監査履歴は未実装である。既存タスクでは、作業作成、承認、取得、完了、失敗、復旧のイベントを記録する予定がある。

ただし、候補を選んだ判断、判断理由、参照データ、提案する次状態、人間に求める判断は、API操作履歴より粒度が細かい。作業取得や完了とは別に、判断を保存する設計が必要である。

### 人間承認の有無

人間承認は、実行前下書きをAIエージェントへ渡す前、内容候補を選ぶ時、使用素材で進めてよいか確認する時、動画生成前確認の時に存在する。投稿可能化、公開、最終完了を個別に止める承認ゲートはまだない。

動画生成工程は、最初の承認ではキューへ入らない。複数箇所の構成と演出案を人間が確認し、「確認用動画を作る」と承認した後に追加される。

### UIで見えている状態・結果・ログ

UIは、依頼、現在の工程、工程ごとの状態、内容候補、使用素材構成、動画生成前確認、生成後レビューを表示する。完成品レビュー用のWeb版Gemini接続、投稿可能化、公開、最終完了の判断はまだない。

### shared type の有無

client、backend、runner は `packages/shared` の工程定義と型を共有している。control plane もこの領域を優先して検討すべきである。

### backend と runner / agent 側の責務分離

backend は実STT、実LLM、Gemini確認、動画生成を実行しない。runner または Codex 側が外部処理を行い、backend には結果と成果物参照を返す方針が既存docsと実装で確認できる。

## 実装設計で具体化する項目

以下はユーザーへの質問リストではなく、仮採用済み方針を前提に次の設計で具体化する項目である。問題があればユーザーが指摘し、意思決定ログを revised にする。

- control plane の状態一覧は、AI処理完了と人間承認済みを分ける前提で具体化する。
- 重要状態は、人間確認待ち、承認済み、却下、修正要求、動画生成要求、投稿可能、公開済み、最終完了を候補にする。
- 最初の人間確認工程は、内容候補整理後、使用素材構成後、動画生成前を優先する。
- 判断ログは、重要なAI判断で判断、理由、根拠参照、提案する次状態、人間に求める判断を含める。
- AIエージェントがLLMやGemini確認を使う場合も、使わない場合も、その判断理由を記録する。
- 動画生成工程は、初回依頼承認とは別の承認ゲートにする。
- UIは、判断要約、理由、根拠参照、次に起きる処理、承認、却下、修正依頼に絞る。
- 品質確認、D4チャット、演出、自動投稿は初期control plane実装には含めない。

## 未解決の懸念

- 現行の動画生成工程は、人間承認が必要な工程として定義されており、候補生成後と動画生成前の承認ゲートは入った。ただし動画生成後の投稿可能化、公開、最終完了の承認ゲートはまだ未実装である。
- 現行の完了状態は、AIエージェントが処理を終えたことを示すだけで、人間が投稿可能または最終完了を承認したことはまだ示さない。
- 現行の成果物参照は、後工程が安全に読むための保存先検証や欠落検出をまだ持たない。
- 重要判断の判断ログは入り始めたが、API操作履歴、失敗復旧履歴、claim履歴はまだ未実装である。
- UIは新方針の内容選択と使用素材確認を最小操作として扱える。ただし確認材料の読みやすさ、内容候補一覧、使用素材構成案、生成動画プレビュー、演出案の中身確認は再設計が必要である。
- claimの所有者、取得時刻、期限切れ復旧は未実装であり、複数AIエージェント運用前に必要である。
- 認証、権限、秘密情報管理は未実装であり、実処理接続前に必要である。
- 品質評価、演出、D4関連を先に入れると、判断記録のない機能追加になる。

## 2026-06-29 作業ログ

- 完成動画レビューは、ユーザーがWeb Gemini結果を貼り付ける作業ではなく、AIエージェントがEdgeでWeb Geminiを使い、結果を成果物として保存する方針に固定した。
- 完成動画画面は、手動の依頼文コピー、レビュー貼り付け、レビュー移し替えを削除し、保存済みレビューの確認と、そのレビューを使った演出再作成だけにした。
- バックエンドは、完成動画に紐づくWeb Geminiレビューを保存・取得し、保存済みレビューから演出作成前の再作成キューを作る処理を追加した。
- 一時ランタイムで、レビュー未取得、レビュー保存、レビュー反映による演出作成前からの再作成キュー作成を確認した。
- EdgeでWeb Geminiは開けたが、Edgeの「Apple EventsからのJavaScriptを許可」が無効のため、AIエージェントによるブラウザ操作は `blocked` として止めた。成功扱いにはしていない。
- Edgeの「Apple EventsからのJavaScriptを許可」が有効になったことを確認した。通常のWeb Geminiレビュー確認は `prepared` まで進み、対象動画と依頼文を保存する。
- EdgeのWeb Geminiへ完成動画を送る処理は外部送信を含むため、準備確認とは分けた。実行する場合は明示的に外部送信実行として扱い、回答を保存して演出作成前の再作成に使う。
- シナリオテストに、完成動画レビュー未取得の確認、Web Geminiレビュー保存、保存済みレビュー取得、レビュー反映による演出作成前からの編集コピー作成、再生成完了の確認を追加した。これにより、完成動画レビューのフィードバックループが壊れた場合は通常シナリオで検出できる。
- 完成動画画面で、Web Geminiレビューがまだ保存されていない場合も、AIエージェントが残したレビュー実行ログを表示するようにした。レビュー準備済み、実行前停止、実行中、保存済み、停止理由、次に必要な操作が見えるため、利用者が「何を待っているのか」を判断できる。
- Web Geminiレビュー実行ログが壊れている場合は、レビュー未取得のように見せず、保存内容が壊れていることをAPIエラーとして返す。画面側はAPIエラー文を表示し、シナリオテストで壊れた実行ログが成功扱いにならないことを確認する。
- 完成動画画面を開いている間は、Web Geminiレビュー状態を定期的に読み直すようにした。AIエージェントがレビュー準備、実行中、保存済みへ進んだ場合に、利用者が再読み込みを押さなくても状態を追える。
- Web Geminiレビュー実行中に失敗した場合は、実行ログを実行中のまま残さず、失敗状態と停止理由へ更新する。画面はレビュー実行失敗として表示し、シナリオテストで失敗ログの読み取りを確認する。
- Web Geminiレビュー本文をファイルから保存する経路でも、空レビューなどで保存に失敗した場合は失敗ログを残す。外部送信なしの一時ランタイムで、空レビュー本文が失敗し、レビュー本体が保存されず、停止理由がログに残ることを確認した。
- Web Geminiレビュー用スクリプト単体の自動テストを追加した。外部送信なしの一時ランタイムで、レビュー本文保存成功時にレビュー本体と保存ログが残ること、空レビュー本文ではレビュー本体を保存せず失敗ログだけを残すことを確認する。
- Web Geminiレビュー本体と実行ログが、現在の完成動画を指しているか確認する処理を追加した。古い完成動画に対するレビューや実行ログは取得・反映できず、現在の動画でレビューを取り直す必要があることを返す。標準テストで古いレビュー本体、古い実行ログ、反映拒否を確認する。
- 完成動画画面から、現在の完成動画に対するWeb Geminiレビュー準備を作り直せるようにした。画面操作は外部送信を実行せず、レビュー対象動画、依頼文、実行ログだけを保存し、AIエージェントが次にWeb Geminiへ送る準備状態を人間が確認できるようにする。
- 標準テストで、レビュー準備APIが現在の完成動画を指す実行ログを作ること、外部送信がまだ必要な状態として残すこと、演出レビュー用の依頼文を作ることを確認した。ブラウザ確認では完成動画画面の「レビュー準備を更新」を押し、準備済み表示が更新されることと画面エラーが出ていないことを確認した。
- 完成動画画面のレビュー準備済み状態から、Web Geminiへ渡す依頼文を折りたたみで確認できるようにした。常時長文を表示せず、必要な時だけ「AIが何を外部レビューへ頼むのか」を読めるようにする。
- Web Geminiレビュー依頼文は、現在の完成動画に合う実行ログが読める時だけ画面へ返すようにした。依頼文ファイルだけが残っている状態では古い依頼文を表示しない。
- Web GeminiレビューをAPIで保存した時も、レビュー本体だけでなく実行ログを保存済みに更新するようにした。後から状態を読んだ時に、準備中ログと保存済みレビューが食い違わないようにする。
- Web Geminiレビュー保存APIで空レビューを受け取った場合も、失敗理由を実行ログへ残すようにした。保存に失敗したのに準備中ログだけが残る状態を避ける。
- 完成動画画面で保存済みWeb Geminiレビューを表示する時、レビューが現在の完成動画に紐づいていること、保存済み状態、次にできる操作、保存日時を小さな状態カードで確認できるようにした。
- 保存済みWeb Geminiレビューがある状態でも「レビューを取り直す」操作を出すようにした。取り直し準備を作った後は古いレビュー本体を消し、画面をレビュー準備済み状態へ戻す。
- Web Geminiレビューを反映して編集コピーを作る時、コピーされた下書きのやり直し理由にレビュー対象動画、レビュー保存日時、演出作成へ渡す改善指示の見出しを残すようにした。
- Web Geminiレビューを反映した後は、元の完成動画側のレビュー実行ログを反映済みに更新し、反映先の編集コピーIDを残すようにした。反映済みレビューの二重反映はAPIと画面の両方で止める。
