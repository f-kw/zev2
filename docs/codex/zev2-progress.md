## 現在フェーズ

Fable案適用前の control plane 最小実装と、工程ごとに評価できるUI再構成の最小実装を確認したフェーズ。

`docs/codex/control-plane-spec.md` の基本方針は、ユーザーから特に反対なしとして仮採用された。現在は、候補生成後と動画生成前でAIエージェントを人間確認待ちに止め、判断ログ、人間確認項目、人間の承認操作を保存し、UIから承認、却下、修正依頼を選べる最小実装が入っている。

次の目的は、ユーザーがUI上で流れと評価ポイントを確認できる状態を保ったまま、ZEVの既存実装を参考にしながら、zev2の責務分離を崩さずに候補生成、Gemini確認、編集案、動画生成の実処理接続へ進むことである。

## ユーザーレビューで判明した問題

### ZC-R-001: 承認画面で、何を見てどう判断すればよいか分からない

確認日: 2026-06-14

見えている問題:

- 現在の承認画面は、AIの判断、理由、根拠、次に起きる処理を表示しているが、それを人間がどう読んで承認、修正依頼、却下を選ぶべきかが分からない。
- 候補生成後の確認と動画生成前の確認は、承認の意味が違う。前者は「候補を映像確認へ進めてよいか」、後者は「編集案を動画生成へ渡してよいか」である。しかし画面上では、その違いが十分に伝わっていない。
- 成果物は表示されているが、承認時にどの成果物を優先して見るべきかが分からない。
- 「承認して続行」を押すと後続で何が動くかは表示されているが、ユーザーの判断責任として何を確認したことになるのかが曖昧である。

人間にとっての影響:

- ユーザーがAIの判断を制御している感覚を持てない。
- 承認、修正依頼、却下の使い分けができない。
- control plane の目的である「AI判断をブラックボックスにしない」状態にまだ届いていない。
- 画面レビュー中に問題点が会話だけで流れると、後続実装で何を直すべきか追えなくなる。

次に必要な設計:

- 承認ゲートごとに、人間が見るべき判断ポイントを定義する。
- 候補生成後の確認では、対象動画、依頼内容、候補区間、候補理由、次に進める価値を確認対象にする。
- 動画生成前の確認では、選ばれた候補、編集案、開始位置、終了位置、テロップ案、生成して確認する価値を確認対象にする。
- 各承認ゲートで、承認、修正依頼、却下を選ぶ条件を人間向けに表示する。
- 承認後に開始される後続処理を、処理名だけではなく「何が起きるか」として表示する。

done 条件:

- 候補生成後の確認画面で、ユーザーが何を確認したら承認できるか分かる。
- 動画生成前の確認画面で、ユーザーが何を確認したら承認できるか分かる。
- それぞれの確認画面で、承認、修正依頼、却下の使い分けが画面上で説明されている。
- 承認時に優先して見る成果物が画面上で分かる。
- 承認後に開始される後続処理が、人間向けの意味で表示されている。
- ユーザーがJSONを読まなくても、承認判断の意味を説明できる。

### ZC-R-003: 承認画面で主な問いと判断材料が同じレベルに見える

確認日: 2026-06-14

見えている問題:

- 画面には依頼、実行状態、次の操作、人間確認、AIの判断、理由、次に起きる処理、根拠、人間の判断、処理の流れが並んでいる。
- しかし、ユーザーが最初に答えるべき主な問いが見えない。
- 現在の表示では、「この編集案で動画生成へ進めてよいか」という問いと、その判断材料である理由、根拠、成果物、処理状態が同じ重さで表示されている。
- `review_required` のような内部状態が、人間向けの問いより目立つ位置に出ている。
- 承認済みになった後も、何を承認した結果なのかが主語として残らず、「承認として記録」という処理結果だけが見えている。

人間にとっての影響:

- ユーザーは、承認欄を見ても「今、自分は何に答えるべきだったのか」を把握しづらい。
- 承認済みの結果を見返しても、「どの問いに対して承認したのか」が分かりづらい。

次に必要な設計:

- 承認画面は、最上位に主な問いを1つ表示する。
- 主な問いの下に、判断するためのサブ情報を置く。
- サブ情報は、確認対象、AIの提案、理由、根拠、承認後に起きる処理、人間の過去判断に分ける。
- 内部状態名は主表示にしない。必要なら補助情報として扱い、人間向けの意味に変換して表示する。
- 承認済みの場合も、「どの問いに対して、どの判断をしたか」を残す。

done 条件:

- 承認画面の先頭に、ユーザーが答えるべき問いが1つ表示されている。
- 候補生成後の確認では、「この候補を映像確認へ進めてよいか」が主な問いとして読める。
- 動画生成前の確認では、「この編集案で動画生成へ進めてよいか」が主な問いとして読める。
- AIの判断、理由、根拠、成果物、処理状態は、主な問いを判断するためのサブ情報として見える。
- 内部状態名が、主な問いや次の操作より強く見えない。
- 承認済みの確認項目でも、どの問いに対して承認したのかが分かる。

### ZC-R-006: JSON表示を開いた後に閉じられず、項目の読み方も分からない

確認日: 2026-06-14

見えている問題:

- JSONを承認判断の補助情報として使うには、項目の意味を同時に参照できる必要がある。
- 現在の表示では、JSON本文だけが出るため、値が何を意味し、承認判断のどこに関係するかが分からない。

人間にとっての影響:

- JSONが判断材料ではなく、読めない内部データに見える。

次に必要な設計:

- JSONを展開した後に閉じる操作を付ける。
- JSON本文の横、または同じ表示領域内に、主要項目の意味を利用者向けに表示する。

- 動画生成計画JSONでは、実素材を切り出したのか、確認用映像にしたのか、使った区間、出力サイズを説明する。

done 条件:

- JSONを展開した状態で、同じ操作位置から閉じられる。
- JSON表示中に、主要項目の意味を同時に確認できる。
- 項目説明は、開いている成果物の種類に合わせて変わる。
- 利用者がJSONの項目名を知らなくても、候補区間、編集案、生成対象、変更理由の意味を把握できる。

## 見つかった現行実装の概要

### タスク状態の定義

現行のAI作業状態は、作業待ち、前工程待ち、実行中、完了、失敗の5種類である。実行前下書きは、下書き、承認済み、却下の3種類である。

これらは処理実行の状態としては最小で成立しているが、候補レビュー待ち、編集案承認待ち、投稿可能、公開済み、最終完了のような人間制御の重要状態はまだ定義されていない。

### 状態遷移の実装

承認済み下書きから直列依存のAI作業が作られる。次の作業取得では、前工程が完了している作業だけが返る。取得済み化すると実行中になり、完了報告で完了になり、失敗報告で失敗になる。

現行の遷移は工程完了の順序を守るためのものであり、AI判断の確認、人間承認、差し戻し、却下、投稿可能化、最終完了を制御するための遷移ではない。

### AIエージェント処理の入口

AIエージェントは、次の作業取得、取得済み化、完了報告、失敗報告のAPIを使う。UIを自動操作して進める構造ではない。

### エージェントの終了条件

dry-run runner は、次の作業取得で作業が返らなくなると終了する。最大処理件数に到達した場合も停止する。

この終了条件は「実行できる作業がない」ことを示すだけで、ユーザー承認済みの最終完了や投稿可能状態を示すものではない。

### 成果物保存の仕組み

現行では成果物本体を状態に埋め込まず、成果物参照を保存する。参照には種類、URI、MIME type、アクセス種別、作成時刻が含まれる。

成果物本体の保存領域、参照検証、欠落時の扱い、後工程の取得方法は後続タスクとして残っている。

### ログの仕組み

現在は詳細な操作ログや監査履歴は未実装である。既存タスクでは、作業作成、承認、取得、完了、失敗、復旧のイベントを記録する予定がある。

ただし、Fable案に必要な判断ログは、API操作履歴より粒度が細かい。候補を選んだ判断、判断理由、参照データ、提案する次状態、人間に求める判断を保存する設計が別に必要である。

### 人間承認の有無

人間承認は、現時点では実行前下書きをAIエージェントへ渡す前に存在する。承認後の各工程では、候補採用、編集案確定、動画生成、投稿可能化、公開、最終完了を個別に止める承認ゲートはない。

動画生成工程は人間承認が必要な工程として扱う意図が型にあるが、現在のdry-run経路では、動画生成まで含める設定で最初の承認を行うと、動画生成工程も後続キューへ入る。

### UIで見えている状態・結果・ログ

UIは、依頼、現在の工程、工程ごとの状態、成功済み工程、失敗工程、実行履歴、成果物件数を表示する。判断ログ、根拠、参照区間、人間承認要求、差し戻し理由の表示はまだない。

### shared type の有無

client、backend、runner は `packages/shared` の工程定義と型を共有している。control plane もこの領域を優先して検討すべきである。

### backend と runner / agent 側の責務分離

backend は実STT、実LLM、Gemini確認、動画生成を実行しない。runner または Codex 側が外部処理を行い、backend には結果と成果物参照を返す方針が既存docsと実装で確認できる。

## 次に調査が必要な箇所

- control plane の状態を既存の作業処理状態に重ねるか、別の人間制御状態として分離するか。
- 判断ログを実行状態と同じ保存先に置くか、監査ログや成果物参照と分けるか。
- 人間承認、却下、差し戻しを、依頼単位、候補単位、工程単位のどこに結び付けるか。
- 動画生成工程の前に必要な承認を、初回依頼承認と分ける方法。
- AIエージェントがLLMやGemini確認を必要と判断した場合に、何を入力、根拠、出力として保存するか。
- UIで最初に出すべき最小情報と、表示しない情報。
- 既存タスク `task-002` から `task-009` の順序に、control plane 設計をどう差し込むか。

## 実装設計で具体化する項目

以下はユーザーへの質問リストではなく、仮採用済み方針を前提に次の設計で具体化する項目である。問題があればユーザーが指摘し、意思決定ログを revised にする。

- control plane の状態一覧は、AI処理完了と人間承認済みを分ける前提で具体化する。
- 重要状態は、人間確認待ち、承認済み、却下、修正要求、動画生成要求、投稿可能、公開済み、最終完了を候補にする。
- 最初の承認ゲートは、候補生成後と動画生成前を優先する。
- 判断ログは、重要なAI判断で判断、理由、根拠参照、提案する次状態、人間に求める判断を含める。
- AIエージェントがLLMやGemini確認を使う場合も、使わない場合も、その判断理由を記録する。
- 動画生成工程は、初回依頼承認とは別の承認ゲートにする。
- UIは、判断要約、理由、根拠参照、次に起きる処理、承認、却下、修正依頼に絞る。
- 品質確認、D4チャット、演出、自動投稿は初期control plane実装には含めない。

## 未解決の懸念

- 現行の動画生成工程は、人間承認が必要な工程として定義されており、候補生成後と動画生成前の承認ゲートは入った。ただし動画生成後の投稿可能化、公開、最終完了の承認ゲートはまだ未実装である。
- 現行の完了状態は、AIエージェントが処理を終えたことを示すだけで、人間が投稿可能または最終完了を承認したことはまだ示さない。
- 現行の成果物参照は、後工程が安全に読むための保存先検証や欠落検出をまだ持たない。
- 重要判断の判断ログは入り始めたが、API操作履歴、失敗復旧履歴、claim履歴はまだ未実装である。
- UIは候補生成後と動画生成前の判断確認に対応したが、確認材料の読み取り、候補一覧、生成動画プレビュー、編集案の中身確認はまだ最小表示である。
- claimの所有者、取得時刻、期限切れ復旧は未実装であり、複数AIエージェント運用前に必要である。
- 認証、権限、秘密情報管理は未実装であり、実処理接続前に必要である。
- Fable案の品質評価、演出、D4関連を先に入れると、判断記録のない機能追加になる。
